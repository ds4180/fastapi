# 🎨 현대적 플랫폼 레이아웃 가이드

> 본 문서는 시스템의 사용자 경험(UX)을 극대화하기 위해 도입된 **3분할 凹(요)자형 레이아웃**과 **내비게이션 정책의 진화 과정**을 정의합니다.

---

## 1. 🏗️ 레이아웃 구조 (3-Panel Core)

시스템은 정보의 계층과 작업 흐름에 따라 세 가지 수직 영역으로 구분됩니다.

1.  **Navigation Bar (좌측)**: 서비스 메뉴, 앱 엔진 목록. ("Explore")
2.  **Main Content (중앙)**: 실제 앱 엔진 구동 구역. ("Execute")
3.  **Context Bar (우측)**: 페이지 보조 도구 및 동적 위젯. ("Assist")

---

## 2. 🧩 내비게이션 정책의 진화 (Evolution of Navigation) 🌟

### 2.1 [하수] 강제 리로드 방식 (Legacy: Full Reload)
- **과거 정책**: 앱 엔진 전환 시 데이터 꼬임을 방지하기 위해 `data-sveltekit-reload`를 사용하여 페이지 전체를 새로고침함.
- **한계**: 화면 전환 시 흰색 깜빡임(White-out)이 발생하고, 매번 리소스를 다시 로드하여 속도가 느림.
- **원인 분석**: 범용 앱 런처(`app/[app_id]`)가 정적인 `const` 변수로 컴포넌트를 로드하여 주소 변화를 감지하지 못했기 때문에 고육지책으로 사용함.

### 2.2 [고수] 반응형 Smart SPA 방식 (Current: Smart SPA)
- **현재 표준**: 앱 런처에 Svelte 5의 **반응성($derived)**을 주입하여 리로드 없이 화면을 즉시 교체함.
- **핵심 메커니즘**:
    1. **컴포넌트 반응성**: `let TargetComponent = $derived(Engines[data.appInfo.main_component])`를 통해 주소창의 `app_id` 변화를 실시간 감지.
    2. **강제 초기화**: `{#key data.appInfo.app_id}` 블록으로 감싸, 앱 종류가 바뀔 때 이전 상태를 완전히 소거하고 새 컴포넌트를 생성.
- **성과**: 화면 전환 속도 비약적 향상 및 플랫폼 본연의 부드러운 앱 UX 완성.

## 3. 🧩 우측 컨텍스트바(Context Bar) 주입 표준

- **방식**: `$lib/store.js`의 `rightSidebar`를 통해 컴포넌트(`DynamicForm` 등)와 메뉴를 동적 주입.
- **클린업**: `$effect`의 리턴 함수를 통해 페이지 이탈 시 상태를 자동 초기화하여 데이터 잔상을 방지.

# 🚀 시스템 App 기반 추상화 아키텍처 가이드 (v1.0)

이 문서는 게시판을 넘어, 시스템의 모든 기능을 독립된 'App'으로 관리하고 실행하기 위해 설계된 **메타데이터 기반 추상화 시스템**의 개념과 구현, 그리고 사용법을 상세히 기술합니다.

---

## 1. 🌟 추상화의 핵심 개념 (The Concept)

### 왜 추상화인가?
기존 방식은 새로운 기능을 추가할 때마다 개발자가 직접 라우트(Route)를 만들고 페이지를 코딩해야 했습니다. 하지만 본 프로젝트의 추상화 시스템은 **"기능(Engine)"**과 **"설정(Metadata)"**을 분리합니다.

### 무엇이 달라졌는가? (The Difference)
- **과거**: 게시판 하나 추가 = 폴더 복사 + 코드 수정 + 빌드 + 수동 메뉴 등록
- **현재**: 게시판 하나 추가 = 관리자 페이지에서 **클릭 한 번으로 앱 엔진 호출**
- **주소 체계**: 은행 사이트와 유사한 `/app/[app_id]/[instance_id]` 구조를 사용하여 통일성과 확장성을 확보했습니다.

---

## 2. 🏗️ 아키텍처 구조 (Architecture)

시스템은 크게 세 단계의 레이어로 작동합니다.

### Layer 1: 데이터베이스 (AppRegistry & Menu)
- **`AppRegistry`**: "이런 엔진(App)이 설치되어 있다"는 장부입니다. (예: `board`, `calendar`)
- **`Menu`**: 특정 앱의 특정 인스턴스(예: 공지사항 게시판)를 가리킵니다. 주소를 직접 저장하지 않고 앱의 ID만 가리키는 지능형 구조입니다.

### Layer 2: 서버 및 통신 (FastAPI & Hooks)
- **지능형 메뉴 로직**: 백엔드가 메뉴 정보를 줄 때, 앱의 설정값(`frontend_route`)을 읽어와서 `/app/board/notice` 같은 최종 URL을 실시간으로 조립합니다.
- **권한 해제**: 앱 메타데이터 조회 API를 전체 공개하여, 로그인 여부와 관계없이 시스템 구조를 파악할 수 있게 합니다.

### Layer 3: 프론트엔드 (Engines & Launcher)
- **컴포넌트 지도 (`lib/index.js`)**: 모든 엔진 컴포넌트들을 이름표를 붙여 관리합니다.
- **범용 실행기 (`routes/app/...`)**: 주소창의 `app_id`를 보고 지도에서 컴포넌트를 찾아 화면에 **"쏙 끼워 넣는(Dynamic Injection)"** 역할을 합니다.

## 3. 🧩 App vs Service 아키텍처 대조 (Advanced)

본 시스템은 두 가지의 핵심 추상화 축을 가집니다. **App (주인)** 과 **Service (부품/플러그인)** 입니다.
구조적으로 두 시스템은 완벽히 대칭을 이루며 레고 블록처럼 조립됩니다. 
*(참고: 테이블명이 `BoardConfig`라서 헷갈릴 수 있으나, 이 테이블은 게시판 전용이 아니라 대시보드 등을 포함하는 **'범용 App 인스턴스 (옵션스토리지)'** 역할을 수행합니다.)*

### 🌟 구조 대조표

| 단계 (계층) | App (메인 플로우) | Service (플러그인 플로우) | 역할 비교 |
|:---|:---|:---|:---|
| **1. 뼈대/카탈로그** | `AppRegistry`<br>("board", "dashboard") | `ServiceRegistry`<br>("comment", "upload") | 시스템에 "이런 종류의 툴킷"이 존재함을 관리자에게 알림 |
| **2. 작동 엔진** | `AppRegistry.main_component` | `ServiceEngine` 테이블<br>("comment_v1" 등 구체적 버전) | 화면을 실제로 그리는 모듈/플러그인과 연결 (버전관리) |
| **3. 인스턴스** | `BoardConfig`<br>(공지사항, 메인대시보드 등) | (Service 자체 인스턴트는 없음)<br>`Binding`의 `custom_config`로 조절 | 실제로 생성된 고유 공간(게시판 1번)과 거기에 필요한 세부 옵션 |
| **4. 사용성 연결** | `Menu` 테이블<br>(메뉴 클릭 → BoardConfig 1번 연결) | `ServiceBinding` 테이블<br>(BoardConfig 1번 ↔ Comment_v1 조립) | "사용자/앱이 실제로 이걸 어떻게 활용하고 띄울 것인가" (조립) |

### 💡 핵심 차이점 (기능적 비대칭)
- **App (주인)**: 혼자서 독립적으로 존재할 수 있고, 고유 URL을 가져서 사용자가 직접 방문합니다 (예: `/app/board/notice`).
- **Service (기생/플러그인)**: 혼자 존재할 수 없고, 반드시 어떤 App 인스턴스(`BoardConfig`)에 **기생(Binding)** 해서만 작동합니다. 공지사항엔 아무것도 안 붙이고, 자유게시판엔 '댓글v1 + 업로드v1'을 붙이고, 갤러리엔 'MinIO업로드v2'를 붙이는 식의 유연한 제어를 가능하게 합니다.

## 4. 📂 파일 변경 및 생성 이력 (Development Log)

### [생성] svelte/src/lib/components/BoardEngine.svelte
- **역할**: 모든 게시판의 UI를 책임지는 '표준 그릇'입니다.
- **특징**: `slug`를 주입받아 스스로 데이터를 로딩하며, 목록/상세/쓰기 모드를 내부적으로 전환합니다.

### [생성] svelte/src/routes/app/[app_id]/[...slug]/
- **역할**: 추상화된 앱들이 실행되는 범용 무대(Viewport)입니다.
- **로직**: `app_id`로 DB를 조회해 어떤 컴포넌트를 띄울지 결정합니다.

### [수정] models.py & migrations
- **`AppRegistry` 테이블 신설**: `frontend_route`, `main_component`, `api_module` 등 추상화 메타데이터를 담는 주머니를 만들었습니다.
- **`Menu` 테이블 확장**: `app_id`, `app_instance_id` 필드를 추가하여 앱 엔진과 직접 연결했습니다.

### [수정] domain/v1/admin/admin_router.py
- **`resolve_menu_url` 추가**: 메뉴 타입이 `APP`인 경우 자동으로 주소를 조립하는 핵심 엔진을 탑재했습니다.

## 5. 🛠️ 사용 방법 (How to use)

### 새로운 App 엔진 등록하기 (개발자용)
1. `lib/components/`에 새로운 엔진(예: `CalendarEngine.svelte`)을 만듭니다.
2. `lib/index.js`에 해당 컴포넌트를 등록합니다.
3. 관리자 페이지(`v1/admin/app`)에서 새 앱을 등록합니다.
   - **main_component**: `CalendarEngine` (이름 정확히 일치)
   - **frontend_route**: `/app/calendar`

### 메뉴에 앱 연결하기 (관리자용)
1. 메뉴 관리(`v1/admin/menu`)로 이동합니다.
2. `메뉴 성격`을 **'🚀 시스템 App 연결'**로 선택합니다.
3. 원하는 앱(예: 게시판 엔진)과 세부 인스턴스(예: 공지사항)를 선택하고 저장합니다.
4. 이제 시스템이 주소를 자동으로 계산하여 연결해 줍니다.

## 6. ⚠️ 주의사항 및 시행착오 (Lessons Learned)

1. **컴포넌트 이름 매칭**: `AppRegistry`에 적는 `main_component` 이름은 `lib/index.js`에서 내보내는(export) 이름과 대소문자까지 완벽히 같아야 합니다.
2. **함수명 불일치 주의**: 엔진 제작 시 `api_module`(`board.api.js`)에 정의된 실제 함수명을 사용해야 합니다. (예: `getPostList`가 아닌 `getBoardPosts`)
3. **Frontend Route 규칙**: 앱 등록 시 `frontend_route`에 `/app/board/[slug]`와 같이 전체 경로 템플릿을 적어주어야 지능형 메뉴가 작동합니다.

## 7. 🚀 v1.5 고도화 업데이트 (2026-03-11)

### 7.1 데이터 기반 동적 URL 해석 (Hardcoding Zero)
- **개선**: 기존 `if app_id == "board"` 방식의 하드코딩 분기문을 완전히 제거했습니다.
- **매커니즘**: `AppRegistry`의 `config_schema` 내에 `instance_info` 메타데이터를 저장하여, 실행 시점에 동적으로 모델을 참조하고 URL을 조립합니다.
  ```json
  "instance_info": {
    "model_name": "BoardConfig",
    "lookup_field": "id",
    "return_field": "slug",
    "placeholder": "[slug]"
  }
  ```

### 7.2 관리 권한 위임 및 보안 이원화
- **이원화 권한**: `min_read_rank`(입구 컷)와 `min_write_rank`(행위 컷)를 독립적으로 설정할 수 있도록 API 및 DB를 고도화했습니다.
- **위임 관리**: `admin_ids` 필드를 통해 시스템 관리자가 아닌 일반 사용자에게도 특정 앱의 관리 주권을 부여할 수 있습니다.

### 7.3 아키텍처 진화 비교 (Legacy vs v1.5)

| 비교 항목 | 기존 방식 (Legacy) | v1.5 고도화 (Data-Driven) | 비고 |
| :--- | :--- | :--- | :--- |
| **확장성** | 새로운 앱 추가 시 **백엔드 코드 수정 필수** | UI에서 **메타데이터(JSON) 등록만으로 즉시 확장** | No-Code 확장 |
| **경로 해석** | `if app_id == "board"` 방식의 하드코딩 분기 | `getattr`을 이용한 **런타임 모델 동적 참조** | 유지보수성 극대화 |
| **보안 모델** | 단일 `min_rank` 체크 (진입 권한만 존재) | **이원화 권한(R/W)** 및 **관리 권한 위임** | 정교한 보안 제어 |
| **연동 방식** | 파편화된 설정 및 수동 경로 조립 | `config_schema` 기반의 **지능형 자동 경로 엔진** | 시스템 지능화 |

> **핵심 가치**: "코드는 시스템의 틀을 만들고, 데이터(JSON)는 시스템의 동작을 결정한다." 이 아키텍처를 통해 개발자의 개입 없이 무한한 기능 확장이 가능한 'Living System'을 지향합니다.

### 7.4 보안 및 권한 통로 최적화 (v1.5.1)
- **이슈**: 범용 실행기(`App Launcher`)가 앱 정보를 가져올 때 관리자 전용 API(`/apps`)를 호출하여 일반 사용자에게 403 에러가 발생하는 문제 발견.
- **해결**: 실행 시점에 **개별 앱 조회 API(`/apps/{app_id}`)**를 호출하도록 전환. 해당 엔드포인트는 조회(Read) 권한만 있으면 누구나 접근 가능하도록 개방되어 보안과 접근성을 동시에 확보함.
- **원칙**: 시스템 메타데이터 조회는 최소한의 권한(Rank 0 이상)으로 허용하되, 실질적인 데이터 조작(Write)은 앱 설정(`min_write_rank`)에 따라 철저히 차단함.

### 7.5 앱 간 내비게이션 및 데이터 무결성 정책 (v1.6.0)
- **문제**: SPA(Single Page Application) 방식의 전환 시, 동일한 범용 실행기 구조 내에서 서로 다른 앱 엔진 간의 데이터 갱신이 지연되거나 멈추는 현상 발생.
- **해결**: 메뉴 링크(사이드바/네비바)에 **`data-sveltekit-reload`** 속성을 강제 적용.
- **정책**: 
    - 인스턴스 앱(게시판 등)과 스태틱 앱(결근계 등) 사이를 이동할 때는 브라우저 수준의 새로고침을 수행하여 모든 상태를 클린하게 초기화함.
    - 이를 통해 데이터 잔상(Ghost Data) 현상을 원천 차단하고 시스템의 안정성을 최우선으로 확보함.
    - 사용자 경험(UX) 딜레이는 `data-sveltekit-preload-data="hover"` 설정을 통해 클릭 전 데이터를 미리 준비함으로써 보완함.

# 🏗️ CMS로의 전환 개요 (CMS Transition Overview)

본 문서는 기존의 '고정형 게시판 서비스'를 **'유연한 CMS(Content Management System) 구조'**로 개조하기 위한 핵심 전략과 설계 방향을 정의합니다.

---

## 🌟 1. 핵심 철학: "모든 것은 보드(Board)다"

- **페이지는 게시글이다**: 독립된 '페이지'(회사소개, 대문 등)는 댓글과 부가 기능이 비활성화된 '특수 게시글'일 뿐입니다.
- **레이아웃은 옵션이다**: 동일한 데이터라도 설정에 따라 '리스트형', '갤러리형', '병렬 뷰(Split View)', '랜딩 페이지'로 변신합니다.
- **기능은 모듈이다**: PWA, Push, WebSocket은 특정 보드에 종속되지 않고, 설정에 따라 ON/OFF 되는 공통 서비스입니다.

---

## 🛠️ 2. 기술적 핵심 전략 (Technical Strategy)

### A. 하이브리드 데이터 모델 (Hybrid Schema)
- **고정 컬럼**: `id`, `title`, `user_id`, `created_at` 등 모든 게시판 공통 항목 (검색/정렬용).
- **동적 컬럼 (JSONB)**: 게시판마다 달라지는 다양한 입력 항목(`extra_data`).
  - 예: 중고장터의 '가격', 업무일지의 '진행률', 프로젝트의 '담당자' 등.
- **TipTap 통합**: 본문 내용은 HTML이 아닌 **JSON 구조**로 저장하여 데이터의 재활용성과 확장성 확보.

### B. 관리자 중심 구조 (Admin-First)
- **코드 수정 없는 운영**: 메뉴 순서, 게시판 생성, 위젯 배치를 관리자 UI에서 즉시 변경.
- **Site Configuration**: 사이트의 전역 설정(로고, 테마, 메인 페이지 ID)을 DB/Redis에서 관리하여 런타임에 반영.

### C. 사내 업무 최적화 레이아웃
- **데스크탑 (Split View)**: Bootstrap 기반의 병렬 레이아웃을 통해 두 개의 문서를 동시에 비교/수정 (Master-Detail 패턴).
- **모바일 (PWA/Push)**: 스마트폰에서는 적층형 레이아웃으로 전환하며, 실시간 Push 알림을 통해 업무 긴급도 대응.

---

## 🔥 3. 독한 피드백 및 개선 과제 (Critical Feedback)

현재 프로젝트의 완성도는 높으나, CMS로 가기 위해 반드시 해결해야 할 병목들입니다.

1.  **도메인 파편화 해결**: `domain/question`, `domain/answer` 등으로 쪼개진 로직을 범용적인 `Content/Post` 도메인으로 통합(Generic CRUD)해야 합니다.
2.  **UI 추상화**: 하드코딩된 `<input>`을 버리고, DB의 메타데이터를 읽어 스스로 폼을 구성하는 **'동적 폼 엔진'**을 Svelte로 구현해야 합니다.
3.  **고급 기능의 옵션화**: WebSocket과 Push 기능을 CMS 설정창의 '체크박스' 하나로 제어할 수 있도록 로직을 연결해야 합니다.

---

## 🚀 4. 단계별 실행 로드맵

1.  **1단계 (기초 공사)**: `BoardConfig` 테이블 생성 및 JSONB 기반의 범용 CRUD API 구축.
2.  **2단계 (에디터/렌더러)**: TipTap 데이터를 완벽하게 소화하는 공통 뷰어 컴포넌트 제작.
3.  **3단계 (관리자 UI)**: 메뉴 관리 및 보드 설정 기능을 갖춘 Admin Dashboard 구축.
4.  **4단계 (레이아웃 엔진)**: 대문(Landing) 및 병렬 뷰(Split View) 레이아웃 스위칭 로직 구현.
5.  **5단계 (통합)**: 기존 PWA, WS, Push 기능을 CMS 설정과 연동하여 실무 투입.

---

> **결론**: 우리는 단순히 게시판을 고치는 것이 아니라, **"코딩 없이도 새로운 업무 도구를 양산할 수 있는 플랫폼"**을 만드는 과정에 있습니다. 이미 준비된 기술 자산(PWA, WS, Redis)을 이 유연한 구조 위에 올리기만 하면 됩니다.

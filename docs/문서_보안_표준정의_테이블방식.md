# 문서 접근 보안 표준 정의 (테이블 기반 M:N 모델)

이 문서는 사용자-그룹 M:N 관계를 기반으로, 문서에 대한 접근 권한을 계층적으로 관리하기 위한 단순화된 보안 모델 및 데이터베이스 스키마 표준을 정의합니다.

## 1. 핵심 원칙

본 표준은 다음 세 가지 핵심 원칙을 기반으로 합니다.

*   **역할 기반 접근 제어 (RBAC)**: 사용자는 부여된 역할(예: 'SystemAdmin', 'DocumentManager', 'BasicUser')에 따라 기본적인 접근 권한을 가집니다.
*   **M:N 사용자-그룹 매핑**: 사용자는 여러 그룹에 속할 수 있으며, 그룹은 여러 사용자를 포함할 수 있습니다.
*   **문서별 명시적 권한 부여**: 기본 역할/그룹 권한 외에, 특정 문서에 대해 개별 사용자나 그룹에게 직접적인 권한을 부여할 수 있습니다.

---

## 2. 데이터베이스 스키마 (SQL - PostgreSQL 예시)

### 2.1. 기본 테이블

| 테이블명                  | 주요 컬럼                                                                  | 설명                                                                                                |
| :------------------------ | :------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------- |
| `users`                   | `user_id` (PK), `username`, `password_hash`, **`role_id` (FK)**, `created_at` | 시스템 사용자 정보. `role_id`를 통해 사용자의 기본 역할을 지정합니다.                                                                 |
| `roles`                   | `role_id` (PK), `role_name`                                                | 시스템에서 정의하는 역할 목록 (예: 'SystemAdmin', 'DocumentManager', 'BasicUser').                                                      |
| `groups`                  | `group_id` (PK), `group_name`                                              | 사용자들을 묶는 그룹 정보 (예: '마케팅팀', 'ProjectX_Team').                                                                 |
| `user_groups`             | `user_id` (FK), `group_id` (FK)                                            | **M:N 연결 테이블**. 사용자와 그룹 간의 다대다 관계를 표현합니다. 한 사용자는 여러 그룹에, 한 그룹은 여러 사용자에 연결될 수 있습니다. |
| `documents`               | `document_id` (PK), `title`, `content`, `owner_user_id` (FK), **`security_level`** | 실제 문서 정보. `owner_user_id`는 문서 소유자를, `security_level`은 문서의 접근 등급('PUBLIC', 'INTERNAL', 'CONFIDENTIAL', 'RESTRICTED')을 나타냅니다. |

### 2.2. 명시적 권한 부여 테이블 (ACL Lite)

`RESTRICTED` 등급의 문서처럼, 특정 사용자나 그룹에게 개별적으로 접근 권한을 부여해야 할 때 사용합니다.

*   **`document_explicit_grants`**:
    *   `grant_id` (PK)
    *   `document_id` (FK)
    *   `grantee_type` (ENUM: 'USER', 'GROUP') - 권한 부여 대상이 사용자인지 그룹인지
    *   `grantee_id` (user_id 또는 group_id)
    *   `permission_level` ('READ', 'WRITE', 'ADMINISTER') - 부여할 권한

---

## 3. 접근 제어 로직

문서 `D`에 대한 사용자 `U`의 특정 작업(`PERMISSION_P`) 접근 권한은 다음 순서로 확인합니다.

1.  **역할 기반 접근**:
    *   `U`의 역할이 `SystemAdmin`이면 **항상 허용**.
    *   `U`의 역할이 `DocumentManager`이고, `D`의 `security_level`이 'INTERNAL', 'CONFIDENTIAL', 'RESTRICTED' 중 하나이며, `D`가 해당 관리자의 업무 범위 내에 있다면 (예: `D.owner_user_id`가 해당 관리자이거나, `D.primary_group_id`가 관리자 속한 그룹이라면), **`READ`/`WRITE` 권한 허용**.

2.  **문서 보안 등급 기반 접근**:
    *   `D.security_level`이 `PUBLIC`이면, 로그인한 사용자에게 `READ` 허용.
    *   `D.security_level`이 `INTERNAL`이면, 로그인한 사용자에게 `READ` 허용.

3.  **그룹 멤버십 기반 접근**:
    *   `U`가 속한 모든 그룹(`user_groups` 테이블 조회)을 가져옵니다.
    *   `document_explicit_grants` 테이블에서 `document_id`=`D.document_id` 이고 `grantee_type`='GROUP' 이며 `grantee_id`가 `U`의 그룹 중 하나이고 `permission_level`이 `PERMISSION_P` (또는 더 높은 권한) 와 일치하는 레코드가 있는지 확인합니다. 있다면 허용.

4.  **직접 권한 부여 확인**:
    *   `document_explicit_grants` 테이블에서 `document_id`=`D.document_id` 이고 `grantee_type`='USER' 이며 `grantee_id`=`U.user_id` 이고 `permission_level`이 `PERMISSION_P` 와 일치하는 레코드가 있는지 확인합니다. 있다면 허용.

5.  **그 외의 경우**: 위에서 허용되지 않았다면 접근 **거부**.

---

## 4. 관리자 접근 권한

*   **`SystemAdmin`**: `users` 테이블에서 `role_id`가 'SystemAdmin' 역할에 해당하는 사용자는 모든 문서에 대한 `ADMINISTER` 권한을 가집니다.
*   **`DocumentManager`**:
    *   `DocumentManager` 역할의 사용자는 `documents.owner_user_id`가 본인이거나, `documents.primary_group_id`가 본인이 속한 그룹에 해당할 경우, 해당 문서에 대한 `ADMINISTER` 또는 `WRITE` 권한을 가질 수 있습니다.
    *   또는, `document_explicit_grants` 테이블을 통해 `DocumentManager` 역할 자체에 특정 문서에 대한 `ADMINISTER` 권한을 부여할 수도 있습니다.

---

## 5. 구현 고려사항

*   **API 엔드포인트**: 문서 조회, 수정, 삭제, 공유 등 각 API 요청마다 위의 접근 제어 로직을 수행합니다.
*   **성능**: 사용자 그룹 멤버십 조회 및 `document_explicit_grants` 테이블 조회 시 인덱스를 활용하여 성능을 최적화합니다.
*   **`security_level` 관리**: 각 문서의 `security_level`은 생성/수정 시 적절하게 설정되어야 하며, 이는 `DocumentManager` 역할 이상이 담당합니다.

# 🕒 리소스 변경 이력 관리 표준 (History & Audit)

> 이 문서는 중요 데이터의 변경 이력을 "누가, 언제, 무엇을, 어떻게" 바꿨는지 완벽하게 추적하기 위한 **이력 관리 아키텍처**를 정의합니다.

---

## 1. 배경 및 목적 (Background & Motivation)
중요한 관리 리소스(게시글, 설정, 결재서류 등)가 변경되었을 때, 과거에 어떤 내용이었는지 대조해야 하는 상황이 빈번합니다. 하지만 단순히 덮어쓰기 방식으로 저장하면 데이터가 증발하여 추적이나 복구가 불가능해집니다.

본 표준은 **Diff/Patch 방식**과 **Soft Delete**를 결합하여 시스템의 모든 변경 이력을 "디지털 타임라인" 형태로 관리하는 데 목적이 있습니다.

## 2. 기존 방식의 문제점 (Analysis of Current State)
- **저장 공간 낭비**: 리소스 하나가 바뀔 때마다 전체 내용을 복사하여 저장하면 DB 용량이 급격히 늘어남.
- **가시성 부족**: "무엇이 바뀌었는가?"를 알려면 두 버전의 텍스트를 눈으로 일일이 대조해야 함.
- **데이터 증발**: 물리적 삭제(`DELETE`) 후에는 "언제 누가 지웠는지"조차 알 수 없음.

## 3. 해결 방안 및 핵심 로직 (Solution & Key Changes)

### 3.1 Diff/Patch 기반 저장 (Efficiency) 🌟
- **핵심**: 전체를 저장하지 않고 **변경된 차이점(Diff)**만 JSONB 객체로 저장합니다. 
- **라이브러리**: `jsonpatch`를 활용하여 `[{ "op": "replace", "path": "/title", "value": "New Title" }]` 와 같은 규격으로 저장합니다.

### 3.2 논리적 삭제 (Soft Delete) 및 버전 관리
- **원칙**: 데이터를 지우지 않고 `is_deleted = true`와 시각(`delete_date`)을 기록하여 "쓰레기통"에 보관합니다. (`[표준]DB_설계_및_명명규칙` 참조)
- **버전**: 수정 시마다 `version` 필드를 1씩 증가시켜 변경의 순서를 확실히 보장합니다.

## 4. 상세 구현 내용 (Implementation Details)

### 4.1 데이터베이스 구조
- **`managed_resource` (예: `post`)**: 최신 데이터와 현재 버전(`version`)을 보관.
- **`resource_history`**: (resource_type, resource_id, version, change_type, changes)
    - `changes`: JSONB 형식의 Diff 리스트 저장.

### 4.2 구현 워크플로우 (Python)
```python
import jsonpatch

def update_resource(resource_id, new_data):
    # 1. 원본 조회
    orig = current_resource.to_dict()
    # 2. Diff 생성
    patch = jsonpatch.make_patch(orig, new_data)
    # 3. History 저장 (patch.patch 리스트를 JSONB로 저장)
    history.save(resource_id, patch.patch)
    # 4. 원본 업데이트 (version += 1)
```

## 5. 성과 및 학습 포인트 (Outcome & Learning)
- **추적성(Audit) 확보**: 1년 전의 변경 사항도 누가 왜 바꿨는지 1초 만에 조회가 가능해졌습니다.
- **데이터 복구**: 의도치 않은 삭제나 수정 발생 시, 과거 이력을 바탕으로 문서를 즉시 원복할 수 있습니다.
- **학습 포인트**: **"데이터 전체가 아닌 관계성(Diff)을 저장"**할 때 얻어지는 저장 효율성과 논리적 견고함을 실습했습니다.

# 🔐 아키텍처: JTI/Redis 기반 세션 관리 및 유지 전략

> 이 문서는 HttpOnly 쿠키와 Redis 기반의 JTI(JWT ID)를 활용한 안전하고 견고한 세션 관리 규격을 정의합니다. 특히 **다중 디바이스 환경에서의 동시 접속 정책**과 **세션 유효성 정합성 유지**에 중점을 둡니다.

---

## 1. 개요 및 철학 (Overview)
본 시스템은 기존 JWT 토큰 기반 방식을 **Redis 기반 JTI(JWT ID) 세션 관리 시스템**으로 전면 전환하여 보안을 강화하고 서버 확장성을 확보합니다. 모든 세션은 `httpOnly` 쿠키로 관리되며, SvelteKit 서버 훅(`handle`, `handleFetch`)을 통해 클라이언트-서버 간 인증 상태의 일관성을 유지합니다.

---

## 2. JTI/Redis 기반 세션 관리 전략

### 2.1 세션 ID (JTI) 발급 및 저장
-   사용자 로그인 성공 시, 백엔드(FastAPI)에서 `uuid4()`를 통해 고유한 `JTI`를 생성합니다.
-   이 `JTI`는 `session_id`라는 이름의 `httpOnly`, `SameSite=Lax`, `Secure` (프로덕션 환경) 쿠키에 담겨 브라우저로 전송됩니다.
-   생성된 `JTI`는 Redis에 `session:{user_id}:{device_category}` 형식의 키로 저장되며, `REFRESH_TOKEN_EXPIRE_DAYS`에 해당하는 만료 시간(TTL)을 가집니다.
-   DB의 `UserSession` 테이블에도 JTI, `user_id`, `device_category`, `login_at` 등의 세션 정보가 기록됩니다.

### 2.2 다중 디바이스 동시 접속 정책 (2개 세션 제한)
-   **디바이스 카테고리 판별:** 백엔드는 로그인 시 요청 헤더의 `User-Agent`를 분석하여 디바이스를 `MOBILE` 또는 `DESKTOP` 카테고리로 분류합니다.
-   **카테고리별 1개 세션:** 한 사용자 아이디당 각 `device_category`별로 하나의 활성 세션만 허용합니다. 즉, 모바일 1대, 데스크톱/태블릿 1대로 총 2개의 세션이 동시에 유지될 수 있습니다.
-   **밀어내기 (Kick-out) 로직:**
    -   동일 `user_id` 및 `device_category`로 새로운 로그인이 발생하면, Redis에서 기존 `JTI`를 삭제합니다.
    -   DB `UserSession` 테이블에서 기존 세션의 상태를 `KICKED_OUT`으로 업데이트하고 `logout_at` 시간을 기록합니다.
    -   새로운 `JTI`를 생성하여 Redis와 DB에 기록하고, 해당 `JTI`를 `session_id` 쿠키로 발행합니다.

### 2.3 세션 유효성 검증 및 동기화
-   **`hooks.server.js` (`handle` 훅):**
    -   페이지 요청 시 브라우저의 `session_id` 쿠키를 읽어 백엔드 `/users/me` API를 호출하여 세션 유효성을 검증합니다.
    -   백엔드에서 401 (Unauthorized) 응답을 받으면, `session_id` 쿠키를 즉시 삭제하고 `event.locals.user`를 `null`로 설정합니다. 이는 **Redis 유실(리부트 등) 또는 Kick-out 발생 시 하이드레이션 불일치를 방지**하고 클라이언트를 즉시 로그아웃 상태로 전환합니다.
-   **`handleFetch` 훅:** 백엔드로 프록시되는 모든 API 요청에 원본 브라우저의 `User-Agent`와 `Cookie` 헤더를 명시적으로 복사하여 전달합니다. 이는 백엔드가 정확한 `device_category`를 판단하고 인증 정보를 활용하는 데 필수적입니다.
-   **Redis 리부트 대응 (`user_crud.py`):**
    -   `get_session_list` (세션 관리 페이지 조회용) 호출 시, DB에 `ACTIVE`로 기록된 세션이라도 Redis에서 해당 `JTI`를 찾을 수 없거나 `JTI`가 일치하지 않으면, DB `UserSession`의 상태를 `EXPIRED`로 업데이트합니다.
    -   이를 통해 Redis 리부트 등으로 Redis 데이터가 유실되어도 DB 세션 목록이 실제 상태와 일관성을 유지합니다.

---

## 3. 프론트엔드 (SvelteKit) 흐름

-   **로그인 (`login/+page.server.js`):** 백엔드 응답의 `Set-Cookie` 헤더에서 `session_id`를 추출하여 SvelteKit의 `cookies.set` API를 통해 브라우저에 직접 설정합니다. (기존 `accessToken`, `refreshToken` 로직 제거)
-   **로그아웃 (`logout/+page.server.js`):** 백엔드 `/api/users/logout`을 호출하여 Redis 세션을 정리하고, 브라우저의 `session_id` 쿠키를 삭제합니다.
-   **세션 모니터링 (`v1/admin/sessions/+page.svelte`):** `/users/sessions` API를 통해 현재 활성 세션 목록을 조회하고, UI에서 Kick-out 기능을 제공합니다. 목록은 최신 로그인 순으로 정렬되며, Redis와 동기화된 정확한 세션 상태를 표시합니다.

## 4. 금기 사항 (Technical Taboos)

-   클라이언트 사이드에서 `session_id` 쿠키를 직접 조작하거나 JTI를 해석하는 행위.
-   백엔드의 `User-Agent` 기반 디바이스 분류 로직을 우회하려는 시도.

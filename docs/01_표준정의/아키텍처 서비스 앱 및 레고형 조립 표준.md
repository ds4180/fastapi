# 🏗️ 아키텍처: 서비스 앱 및 레고형 조립 표준

> 이 문서는 시스템의 모든 기능을 **"지능형 본체"**와 **"탈부착식 서비스"**로 구분하여 조립하는 레고형 아키텍처의 핵심 규격을 정의합니다.
본 시스템은 모든 기능을 **"지능형 본체"**와 **"탈부착식 서비스"**로 구분하며, 특히 모든 기능을 **App Registry**에 등록하여 **ID 기반으로 호출**하는 시스템 제어 방식을 채택합니다.

---

## 2. 앱의 분류: 인스턴스형 vs 논-인스턴스형
모든 애플리케이션은 사용 목적에 따라 두 가지 형태로 분류되어 `AppRegistry`에서 통합 관리됩니다.

### 2.1 인스턴스형 앱 (Instance-based App)
- **정의**: 하나의 소스 코드로 여러 개의 독립된 방(인스턴스)을 생성하는 앱.
- **예시**: 게시판(Board), 페이지(Page).
- **관리**: `BoardConfig` 등의 인스턴스 설정과 `ServiceBinding`(레고형 서비스)을 통해 각기 다른 성격을 가집니다.

### 2.2 논-인스턴스형 앱 / 일반 앱 (Non-Instance / Static App) 🌟
- **정의**: 시스템 내에서 단 하나만 존재하며 고정된 경로를 가진 유일무이한 기능.
- **예시**: 대시보드, 시스템 설정, 로그 분석기, 사용자 관리.
- **관리**: `AppRegistry`에 **풍부한 메타데이터(Rank, Admin, Description)**를 등록하여 시스템적으로 보호합니다.

---

## 3. 앱 명부 상세 규격 (App Registry Specification) 📚
모든 앱은 관리자화면의 '앱 등록' 기능을 통해 아래의 명세를 등록해야 합니다.

1. **식별 정보**: `app_id` (고유값), `title` (공식 명칭), `description` (타이틀 하단 요약 설명)
2. **권한 이원화**:
   - `min_read_rank`: 메뉴 노출 및 데이터 조회 가능 등급 (입구/조회 보안)
   - `min_write_rank`: 데이터 생성, 수정, 삭제 가능 등급 (행위/변경 보안)
3. **관리 권한 분리 (Delegated Admin)**:
   - `admin_ids` (JSONB List): 해당 앱의 설정권을 가진 사용자 ID 목록. (시스템 관리자가 아니어도 운영 가능)
4. **레이아웃 유연성 (Display Modes)** 🎨
   - `BLOCK Mode`: 컨테이너 너비를 100% 사용하여 위아래로 적층되는 레이아웃. (예: 게시판 하단 댓글)
   - `INLINE/GRID Mode`: 대시보드나 분할 화면의 특정 그리드 칸(1x1, 2x1 등)에 맞춰 렌더링되는 레이아웃. 이게 바로 **'위젯'**의 본체임.

---

## 4. 작동 매커니즘 (Key Mechanisms)
- **정의**: 앱(게시판, 페이지 등)이 정상 작동하기 위해 반드시 알고 있어야 하는 **'생존 설정'**들입니다.
- **포함 항목**: 슬러그(Slug), 레이아웃(List/Gallery), 에디터 종류, 페이징 수, **읽기 권한(Menu Rank)** 등.
- **특징**: "내가 누구인가"를 결정하는 메타 서비스는 해당 앱의 설정(`BoardConfig`) 안에 내포됩니다.

### 1.2 레고형 서비스 (Lego Service App) 🧱
- **정의**: 본체의 기능과는 무관하지만, 필요에 따라 앱의 능력을 확장해주는 **'탈부착 기능'**들입니다.
- **포함 항목**: 댓글(Comment), 업로드(Upload), 추천(Reaction), 태그(Tag), 설문(Survey), 투표(Poll) 등.
- **특징**: "있어도 되고 없어도 되는" 독립적 엔진입니다. `ServiceBinding` 테이블을 통해 딸깍! 하고 결합됩니다.

---

## 2. 권한 및 보안 원칙 (Entry vs Action)

사용자님의 **"입구 컷"**과 **"행위 컷"** 논리를 다음과 같이 표준화합니다.

1. **읽기 권한 (Navigation Security)**:
   - **위치**: 메뉴(`Menu`) 테이블.
   - **로직**: 사용자가 메뉴를 볼 수 있다면 해당 앱의 데이터를 읽을(조회할) 권리가 있는 것으로 간주합니다. (입구에서 검증 완료)
2. **쓰기 권한 (Action Security)**:
   - **위치**: 서비스 바인딩(`ServiceBinding`) 테이블.
   - **로직**: 글쓰기, 댓글쓰기 등 데이터 변경 행위는 바인딩 시점에 설정된 `min_write_rank`를 기준으로 검증합니다. (행위 시점에 검증)

---

## 3. 데이터 조립 매커니즘 (Lego Binding Mechanism)

### 3.1 서비스 바인딩 (Service Binding) 🔗
- **매커니즘**: 메인 앱 인스턴스(예: 공지게시판)가 생성되면, 원하는 서비스 엔진(`engine_id`)을 바인딩 테이블에 등록합니다.
- **버전 관리**: 바인딩 시 `v1`, `v2` 등의 **버전 정보**를 고정하여, 시스템 업데이트로 인한 데이터 오작동을 방지합니다.

### 3.2 조립 로직 (Assembly Logic)
1. **Request**: 사용자가 게시판 진입.
2. **Backend**: `BoardConfig`(메타 서비스)와 연결된 `ServiceBinding`(레고 조각들) 정보를 한 패키지로 응답.
3. **Frontend**: 수신된 엔진 목록을 순회하며 동적 컴포넌트(`<svelte:component>`)로 UI를 바닥부터 천장까지 조립해 나감.

---

## 4. 기대 효과 (The Impact)

- **재사용성(Reusability) 극대화**: "댓글 엔진" 하나를 잘 만들어두면, 게시판뿐만 아니라 모든 앱에 **'재탕(재사용)'** 가능.
- **유지보수 간결화**: 특정 기능을 수정하고 싶을 때, 전체 시스템을 건드리지 않고 해당 **'레고 엔진'**만 교체하거나 패치 가능.
- **데이터 정합성**: 모든 데이터가 자신의 출처(엔진 버전)를 기억하므로 데이터 역사가 명확하게 보존됨.

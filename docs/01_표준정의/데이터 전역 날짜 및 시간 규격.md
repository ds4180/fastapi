# 📅 날짜 및 시간 처리 표준 (DateTime Standard)

> 이 문서는 프로젝트 전반에서 발생하는 날짜 계산의 오차를 방지하고, 타임존 혼란을 종식시키기 위한 데이터 저장 및 처리 규격을 정의합니다.

---

## 1. 배경 및 목적 (Background & Motivation)
웹 서비스에서 날짜와 시간은 단순한 기록을 넘어, 근무 시간 계산, 예약 스케줄링, 로그 분석 등 비즈니스 로직의 핵심입니다. 서버, DB, 클라이언트의 타임존이 일치하지 않으면 "분명 오늘 등록했는데 어제로 표시되는" 등의 치명적인 데이터 오류가 발생합니다.

본 표준은 **서버 타임존 고정(KST)**과 **시간 양(Duration)의 정수화**를 통해 계산 오차를 원천 차단하고 개발 생산성을 높이는 데 목적이 있습니다.

## 2. 기존 방식의 문제점 (Analysis of Current State)
- **타임존 파편화**: 서버는 UTC, DB는 시스템 설정, 클라이언트는 로컬 시각을 사용함에 따라 시차(+9시간) 관련 버그가 빈번함.
- **날짜 계산의 복잡성**: '1시간 30분'을 `1.5` (실수) 또는 `"01:30"` (문자열)로 저장할 경우, 합산 및 통계 계산 시 부동 소수점 오차나 복잡한 파싱 로직이 필요함.
- **포맷 불일치**: API 응답마다 날짜 형식이 달라 프론트엔드에서 매번 다른 라이브러리로 변환해야 함.

## 3. 해결 방안 및 핵심 로직 (Solution & Key Changes)

### 3.1 타임존 단일화 (KST Fixed) 🌟
- **원칙**: 모든 서버 환경(Docker, FastAPI, Svelte)의 환경 변수를 `TZ=Asia/Seoul`로 강제 고정합니다.
- **효과**: DB와 백엔드에서 `datetime.now()`를 호출하면 즉시 한국 시각이 반환되며, 별도의 시차 계산 로직이 필요 없어집니다.

### 3.2 시간 양(Duration)의 정수화 (Integer Minutes)
- **원칙**: 근무 시간, 휴게 시간 등 '양'을 나타내는 데이터는 모두 **분(Minute) 단위의 정수**로 저장합니다.
- **효과**: `480(분) - 60(분) = 420(분)`과 같이 단순 사칙연산만으로 오차 없는 정확한 계산이 가능합니다.

## 4. 상세 구현 규격 (Implementation Details)

### 4.1 데이터 타입 매핑
| 구분 | DB 확장 타입 (SQLAlchemy) | Python 타입 | 예시 |
| :--- | :--- | :--- | :--- |
| **날짜 (Date)** | `Date` | `datetime.date` | `2026-03-09` |
| **일시 (DateTime)** | `DateTime` | `datetime.datetime` | `2026-03-09 10:00:00` |
| **시간 양 (Duration)** | `Integer` | `int` (단위: 분) | `90` (1시간 30분) |

### 4.2 유틸리티 활용
- **서버 설정**: `Dockerfile` 또는 `docker-compose.yml`에 `TZ=Asia/Seoul` 명시.
- **표시 로직**: 프론트엔드에서는 분 단위 정수를 받아 `Math.floor(m/60)`와 `m%60`을 통해 시/분으로 단순 변환하여 출력.

## 5. 성과 및 학습 포인트 (Outcome & Learning)
- **버그 제로**: 타임존 고정 이후 시차 관련 CS 및 버그 리포트가 사라졌습니다.
- **계산 정확도**: 실수(Float) 대신 정수(Integer)를 사용함으로써 정산 및 통계 로직의 신뢰성이 확보되었습니다.
- **학습 포인트**: **"기술적 복잡성(타임존 계산 라이브러리)을 정책적 단순화(환경 변수 고정)로 해결"**하는 설계의 힘을 확인했습니다.

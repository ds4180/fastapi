# 🛡️ 시스템 권한 및 보안 상위 표준 (Global Security Standard)

> 이 문서는 본 프로젝트의 모든 보안 정책의 근간이 되는 **상위 표준**입니다. 모든 하위 보안 문서(리소스 보안, 파일 보안 등)는 본 문서에서 정의한 Rank 체계를 따릅니다.

---

## 1. 전역 권한 레벨 (Global Rank Levels)
시스템 전체의 접근 권한은 0부터 4까지의 정수형 **Rank**로 관리합니다.

| 레벨 | Role 명칭 | 설명 | 비고 |
| :--- | :--- | :--- | :--- |
| **0** | `GUEST` | 비로그인 / 단순 방문자 | 공개 데이터만 접근 가능 |
| **1** | `USER` | 일반 사용자 (정회원) | 본인 소유 리소스의 CRUD |
| **2** | `STAFF` | 업무 담당 실무자 | 특정 도메인의 관리 권한 보유 |
| **3** | `MANAGER` | 책임 관리자 | 시스템 설정 및 비공개 페이지 접근 |
| **4** | `ADMIN` | 최고 관리자 | **Master Key.** 모든 보안 로직 우회 |

---

## 2. 보안 주권 원칙 (Sovereignty of App Authority)

본 프로젝트의 모든 보안 권한의 **원천(Source of Truth)**은 `AppRegistry`입니다.

### 2.1 앱 주권 위계 (Authority Hierarchy)
1. **App (Sovereign)**: 앱 등록 시점에 부여된 `min_rank`가 해당 기능의 **절대적 보안 등급**을 결정합니다.
2. **Menu (Reflector)**: 메뉴는 독자적인 권한을 갖지 않으며, 연결된 앱의 권한을 상속받아 유저에게 **'보여줄지 말지(Visibility)'**만 결정합니다.
3. **Waterfall Mechanism**: 앱의 권한이 바뀌면, 그 앱을 가리키는 모든 메뉴와 링크의 권한이 **폭포수처럼 자동으로 하향 전파**됩니다.

### 2.2 메뉴 노출 제어 (Navigation Visibility)
- **대상**: 서비스 전체 진입권.
- **매커니즘**: 서버가 메뉴 트리를 구성할 때, 각 메뉴에 바인딩된 `app_id`의 `min_rank`를 `AppRegistry`에서 실시간으로 조회하여 유저의 Rank와 비교합니다.
- **원칙**: 메뉴 노출 여부는 보안의 **편의성**을 위한 것이며, 실제 보안의 **완결성**은 앱 호출 단계에서 결정됩니다.

---

## 3. 앱 주권 보안의 이점
1. **단일 관리 포인트**: 수백 개의 메뉴를 고칠 필요 없이, 앱 명부 한 곳에서 전체 시스템의 등급을 조정할 수 있습니다.
2. **보안 구멍 원천 봉쇄**: 메뉴 설정 실수로 노출된 링크라 할지라도, 앱 자체의 보안 레벨이 높으면 진입 자체가 불가능합니다. (이중 방어벽)
3. **일관성 유지**: 단순 페이지부터 복잡한 모듈까지 모두 동일한 `app_id` 체계 안에서 관리됩니다.

---

## 3. 권한 이원화 원칙 (Authority Dualization) 🌟
본 프로젝트는 설정의 간결함과 보안의 명확성을 위해 **"읽기는 입구에서, 쓰기는 안에서"** 검증하는 이원화 원칙을 채택합니다.

1. **읽기(Access)**: [메뉴 설정] 기반. 메뉴의 `min_rank`를 통과하면 해당 서비스의 조회가 가능함. 별도의 게시판별 읽기 권한을 중복 설정하지 않음을 지향합니다.
2. **쓰기(Action)**: [인스턴스 설정] 기반. 인스턴스(예: `BoardConfig`) 생성 시점에 명시한 `min_write_rank`를 기준으로 글쓰기 버튼 활성화 및 API 저장을 제어합니다.

예) **공지게시판**: 메뉴 `min_rank=2` (모든 회원 읽기 가능) / 게시판 `min_write_rank=3` (관리자만 작성 가능)

---

## 3. 용어의 정의 (Terminology)
문서 간 혼선을 방지하기 위해 다음 용어를 통일합니다.
- **관리 리소스 (Managed Resource)**: `Post`, `DayOff`, `Alert` 등 데이터베이스에서 관리되는 모든 의미 있는 데이터 단위. (기존 '문서'라는 용어를 대체)
- **파일 자산 (File Assets)**: 업로드된 물리적 파일. (`[표준]문서_파일_관리` 참조)

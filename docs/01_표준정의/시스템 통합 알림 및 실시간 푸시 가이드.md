# 🔔 전역 알림 및 푸시 시스템 표준 (Notification Standard)

> 이 문서는 단순한 알림을 넘어, **업무 이행 보장 및 책임 소지 명확화**를 위한 다단계 알림 시스템의 작동 원리와 규격을 정의합니다. 웹소켓(WebSocket)과 PWA 푸시를 결합하고, 관리자용 알림 생성/관리 기능을 포함합니다.

---

## 1. 배경 및 목적 (Background & Motivation)

기존의 단방향 공지사항은 사용자가 "못 봤다"고 변명하거나, 중요도가 낮은 알림에 묻혀 긴급 지시사항이 전달되지 않는 문제가 있었습니다. 

본 표준은 **알림의 강제성(Force Level)**을 5단계로 세분화하고, 웹소켓(WebSocket)과 PWA 푸시를 결합하여 **어떤 상황에서도 메시지가 도달하고 확인되도록** 하는 데 목적이 있습니다. 더불어, 관리자가 이러한 알림을 효율적으로 생성하고 관리할 수 있는 기능을 제공합니다.

## 2. 기존 방식의 문제점 (Analysis of Current State)

-   **전달 확인 불가**: 공지를 올렸으나 실제로 누가 읽었는지, 읽는 척만 했는지 알 수 없음.
-   **실시간성 부족**: 페이지를 새로고침해야만 공지가 나타나는 수동적 방식.
-   **알림 피로도**: 중요하지 않은 성공 메시지(저장 완료 등)와 긴급 점검 공지가 같은 형태로 노출되어 사용자 경험 저하.
-   **관리 기능 부재**: 관리자가 알림을 생성, 수정, 삭제, 활성/비활성 할 수 있는 UI가 부족함.

## 3. 해결 방안 및 단계별 로직 (Solution & Key Changes)

### 3.1 5단계 알림 등급 (The 5-Level Force) 🌟

| 등급 | 명칭 | 형태 | 핵심 동작 | 용도 |
| :--- | :--- | :--- | :--- | :--- |
| **L1** | **일반 알림** | Toast | 5초 후 자동 소멸 | 저장/삭제 성공 안내 |
| **L2** | **상태 배너** | Banner | 해결 전까지 상단 고정 | 서버 점검 예고, 장애 상황 |
| **L3** | **일반 확인** | Modal | 클릭 전까지 화면 차단 | 규정 변경, 중요 공독 |
| **L4** | **강제 지시** | Timed Modal | **대기 시간 필수 + 클릭 패널티** | 긴급 안전 지시, 배차 공지 |
| **L5** | **시스템 락** | Full Screen | 전체 기능 차단 | 비인가 접근, 전면 점검 |

### 3.2 Level 4 "인과응보" 타이머 ⏳

-   **로직**: 확인 버튼에 5~10초의 카운트다운을 부여합니다.
-   **패널티**: 시간이 다 되기 전 클릭 시 **타이머가 초기화**되어 처음부터 다시 읽게 만듭니다. (정독 유도)

### 3.3 알림 관리자 기능 (Admin Management) 🆕

-   **목표**: 관리자가 알림을 생성, 조회, 수정, 삭제, 활성/비활성 처리할 수 있는 통합 관리 페이지를 제공합니다.
-   **백엔드 API**: `GET /alert/list`, `POST /alert/create`, `DELETE /alert/delete/{alert_id}`, `POST /alert/toggle/{alert_id}` 엔드포인트 활용.
-   **프론트엔드 UI**: `/v1/admin/alerts` 경로에 전용 관리 페이지를 구축하여 CRUD 기능을 제공합니다. 기존 디자인 표준(카드, 반응형, Iconify)을 따릅니다.

## 4. 상세 구현 가이드 (Implementation Details)

### 4.1 하이브리드 전파 (Hybrid Delivery)

-   **온라인**: 웹소켓(`manager.broadcast`)을 통해 0.1초 만에 브라우저 UI 갱신.
-   **오프라인/백그라운드**: PWA 웹 푸시(`Web Push API`)를 통해 스마트폰 알림창으로 전송.

### 4.2 데이터 스키마 (JSON)

```json
{
  "id": "alert_20260309_001",
  "level": 4,
  "style": "danger",
  "message": "긴급: 1번 출구 폐쇄. 우회로를 이용하십시오.",
  "reset_sec": 7,  // L4 전용 대기 시간
  "confirm_text": "지시 사항 확인함",
  "redirect_url": "/map/exit-1"
}
```

### 4.3 프론트엔드 컴포넌트 (`Alert.svelte`) 역할

-   `Alert.svelte`: 클라이언트 측에서 `GET /alert/active` API를 주기적으로 폴링하여 현재 활성화된 알림 목록을 가져옵니다.
    -   `L1`, `L2` 알림은 Toast 또는 Banner 형태로 사용자에게 노출합니다.
    -   `L3` 알림은 모달 형태로 화면을 차단하고 사용자 확인을 유도합니다.
    -   `L4`, `L5` 알림은 고도화된 UI/UX(타이머, 전면 락)를 통해 사용자에게 강제적으로 메시지를 인지시킵니다.
-   `alertState`: 알림 목록, 해제된 알림 ID, 방해 금지 모드(`isQuietMode`) 등을 관리하는 Rune 스토어.

## 5. 성과 및 학습 포인트 (Outcome & Learning)

-   **업무 기강 확립**: "확인함" 버튼 클릭 로그를 DB에 남겨 업무 과실 방지 기반을 마련했습니다.
-   **실시간 반응성**: 웹소켓과 폴링(Polling)의 지능형 결합을 통해 네트워크 불안정 상황에서도 알림 누락을 최소화했습니다.
-   **관리 효율성 증대**: 관리자 페이지를 통해 알림 생성 및 관리가 용이해졌습니다.
-   **학습 포인트**: 사용자의 **심리적 요인(정독 유도 패널티)**을 로직에 반영하고 관리 기능을 더할 때 시스템의 실효성이 얼마나 높아지는지 확인했습니다.

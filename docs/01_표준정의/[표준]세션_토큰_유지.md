# 🔐 로그인 및 세션 유지 표준 (Authentication & Session)

> 이 문서는 httpOnly 쿠키 기반의 안전한 세션 관리와 **SPA 환경에서의 경쟁 상태(Race Condition) 방지**를 위한 최신 규격을 정의합니다.

---

## 1. 개요 및 철학 (Overview)
본 시스템은 보안 강화를 위해 모든 토큰을 `httpOnly` 쿠키로 관리하며, SvelteKit의 서버 훅(`Hooks`)을 통해 클라이언트-서버 간 인증 정합성을 유지합니다. 특히 **SPA(Single Page Application)** 환경에서 빈번하게 발생하는 API 호출 중 토큰 갱신 꼬임 현상을 방지하는 데 중점을 둡니다.

---

## 2. 이중 방어 시스템 (The Dual-Gate System)

### 2.1 제1관문: 선제적 갱신 (`handle` Hook)
- **역할**: 페이지 진입 시점에 `access_token`이 만료되었으나 `refresh_token`이 살아있는 경우, 백엔드 API를 호출하여 세션을 미리 연장합니다.
- **효과**: 사용자가 페이지를 보는 동안 흐름이 끊기지 않도록 미리 조치합니다.

### 2.2 제2관문: 데이터 통신 중 복구 (`handleFetch` Hook) 🌟
- **역할**: 실제 API 통신 도중 401(Unauthorized)이 발생하면, 요청을 일시 정지하고 즉시 토큰을 리프레시한 뒤 원래의 요청을 재시도(Re-try)합니다.
- **SPA 최적화 (v1.9.1)**: 
    - **무한 루프 방지**: 리프레시 API 호출 자체는 훅 로직에서 제외하여 재귀적 호출을 차단합니다.
    - **경쟁 상태 제어**: 리프레시 도중 발생하는 다른 401 요청들이 세션을 파괴하지 않도록 가드를 둡니다.

---

## 3. SPA 환경에서의 세션 정합성 수칙

1.  **리프레시 가드**: `handleFetch` 내부에서 리프레시 API 경로를 반드시 체크하여 별도 처리해야 합니다.
    ```javascript
    if (request.url.includes('/users/refresh')) return await fetch(request);
    ```
2.  **데이터 클로닝**: 재시도 요청 시 `request.clone()`을 사용하여 원본 요청 정보를 손실 없이 복구해야 합니다.
3.  **로그아웃 가역성**: `/user-logout` 등 사용자가 의도적으로 이동하는 경로에서는 강제 세션 파괴가 일어나지 않도록 예외 처리합니다.

---

## 4. 금기 사항 (Technical Taboos)

- **리프레시 실패 시 즉각 폭파 금지**: 단순 네트워크 오류로 인한 리프레시 실패가 즉시 로그아웃으로 이어지지 않도록, 인증 필수 API(`users/me`) 실패 시에만 세션을 종료할 것을 권장합니다.
- **event.fetch와 native fetch 구분**: 훅 내부에서 리프레시를 시도할 때는 불필요한 훅 재발동을 막기 위해 신중하게 사용해야 합니다.

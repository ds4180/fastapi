# 🔐 로그인 및 세션 유지 표준 (Authentication & Session)

> 이 문서는 수많은 시행착오 끝에 정립된 **httpOnly 쿠키 기반의 안전한 세션 관리** 프로젝트 전용 가이드입니다. 

---

## 1. 배경 및 목적 (Background & Motivation)
초기 프로젝트는 자바스크립트로 토큰을 다루는 방식(LocalStorage 등)을 채택했으나, 보안 이슈(XSS 공격 취약성)가 제기되었습니다. 이에 따라 브라우저의 자바스크립트가 접근할 수 없는 **`httpOnly` 쿠키** 기반의 인증 체계로 전면 전환하게 되었습니다.

본 표준은 **SvelteKit Server Hooks**를 사용하여, 보안 강화 이후 발생한 "서버와 클라이언트 간의 토큰 교환 오류"를 완벽히 해결하는 데 목적이 있습니다.

## 2. 기존 방식의 문제점 (Analysis of Current State)
- **보안의 벽 (httpOnly)**: JS가 쿠키를 읽지 못해 프론트엔드에서 수동으로 토큰을 헤더에 넣는 방식이 불가능해짐.
- **SSR 인증 유실**: 서버 사이드 렌더링(SSR) 시, 표준 `fetch` 요청은 사용자의 세션(Cookie) 정보를 자동으로 전달하지 않아 "새로고침 시 로그아웃되는 현상" 발생.
- **토큰 갱신 실패**: 액세스 토큰 만료 시, 수동으로 갱신 로직을 모든 페이지에 넣을 수 없어 사용자 경험이 저하됨.

## 3. 해결 방안 및 핵심 로직 (Solution & Key Changes)

### 3.1 이중 방어 시스템 (The Dual-Gate System) 🌟
- **관문(Handle Hook)**: 사용자가 페이지를 열기 **직전**에 서버 단에서 쿠키를 확인하고, 만료되었다면 즉시 토큰을 갱신(Refresh)하여 요청을 지속합니다.
- **안전장치(HandleFetch Hook)**: 데이터 통신 **도중**에 401 에러가 발생하면, 요청을 일시 정지하고 토큰을 발급받은 뒤 다시 시도(Re-try)하여 사용자에게 에러를 노출하지 않습니다.

### 3.2 서버 전용 통신 (`event.fetch`)
- **원칙**: 모든 서버 사이드 로직에서는 표준 `fetch` 대신 SvelteKit이 제공하는 `event.fetch`를 사용합니다. 이 레퍼가 사용자의 세션 정보를 백엔드에 자동으로 운반합니다.

## 4. 상세 구현 가이드 (Implementation Details)

### 4.1 핵심 파일: `hooks.server.js` (관문 엔진)
```javascript
/**
 * 모든 서버 요청을 가로채서 세션을 관리하는 핵심 훅
 * @type {import('@sveltejs/kit').Handle}
 */
export const handle = async ({ event, resolve }) => {
    // 1. 쿠키에서 토큰 추출
    // 2. 만료 시 백엔드 /api/refresh 호출하여 쿠키 갱신
    // 3. 갱신된 정보로 사용자 정보(Rank 등)를 event.locals에 주입
    return await resolve(event);
};
```

### 4.2 금기 사항 (Taboos)
- **자바스크립트 직접 수정(api.js)**: 브라우저 단에서 토큰을 가공하려 하지 마십시오.
- **Try-Catch 속 리다이렉트**: SvelteKit의 `redirect` 에러를 삼키지 마십시오. (인증 전파 방해)

## 5. 성과 및 학습 포인트 (Outcome & Learning)
- **최고 수준의 보안**: 자바스크립트 공격(XSS)으로부터 토큰을 완벽하게 보호하면서도 매끄러운 세션을 유지합니다.
- **유연한 확장성**: 백엔드의 인증 방식이 바뀌더라도 `hooks.server.js` 한 곳만 수정하면 전체 페이지에 즉시 적용됩니다.
- **학습 포인트**: **"보안(Security)과 편의(UI/UX)는 트레이드오프 관계지만, 아키텍처(Hooks)를 통해 이 둘을 양립시킬 수 있음"**을 증명했습니다.

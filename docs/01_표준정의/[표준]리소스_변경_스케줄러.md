# 📅 리소스 변경 스케줄러 표준 (Resource Scheduler)

> 이 문서는 중요 리소스의 변경 사항을 미래의 특정 시점에 자동으로 적용하기 위한 **스케줄러 아키텍처 및 구현 표준**을 정의합니다.

---

## 1. 배경 및 목적 (Background & Motivation)
근로계약 내용의 변경, 공지사항의 예약 게시, 임시 권한의 자동 회수 등은 작업 시점과 적용 시점이 다를 수 있습니다. 수동으로 날짜에 맞춰 업데이트하는 것은 실수하기 쉽고 운영 부담이 큽니다.

본 표준은 **FastAPI Lifespan**을 활용한 비동기 백그라운드 루프를 구축하여, 별도의 복잡한 인프라(Cron, Celery 등) 없이도 **미래 시점의 자동 업데이트**를 구현하는 데 목적이 있습니다.

## 2. 기존 방식의 문제점 (Analysis of Current State)
- **조회 시 적용 (Lazy Apply)**: 사용자가 볼 때 업데이트하는 방식은 첫 번째 방문자의 응답 지연을 초래함.
- **외부 의존성**: Cron이나 Celery는 소규모 프로젝트에서 설정 및 관리 복잡성이 큼.
- **실패 시 복구 불가**: 작업 도중 서버가 꺼지거나 오류가 나면 어떤 작업이 누락되었는지 확인하기 어려움.

## 3. 해결 방안 및 핵심 로직 (Solution & Key Changes)

### 3.1 비동기 백그라운드 루프 (Lifespan Scheduler) 🌟
- **핵심**: FastAPI의 `lifespan` 이벤트 내에서 `asyncio.create_task()`를 사용하여 앱 실행과 동시에 무한 루프를 돌립니다.
- **주기적 조회**: 60초마다 `scheduled_changes` 테이블을 감시하여 시각이 도달한 항목을 처리합니다.
- **원자성 보장**: 작업 하나하나를 독립적인 트랜잭션으로 처리하여, 특정 작업 실패가 전체 루프를 중단시키지 않게 설계했습니다.

## 4. 상세 구현 가이드 (Implementation Details)

### 4.1 데이터 스키마 (`ScheduledChange`)
| 컬럼명 | 타입 | 설명 |
| :--- | :--- | :--- |
| **`resource_type`** | `String` | 대상 모델 (POST, USER 등) |
| **`resource_id`** | `Integer` | 대상 리소스 ID |
| **`effective_date`**| `DateTime`| **변경 적용 예약 시각** |
| **`changes`** | `JSONB` | 적용할 Diff 데이터 (jsonpatch) |
| **`status`** | `Enum` | 작업 상태 관리 (PENDING, APPLIED) |

### 4.2 스케줄러 엔진 예시 (Python)
```python
# scripts/utils/scheduler.py
async def scheduler_loop():
    while True:
        # 1. 시각 도달 PENDING 작업 조회
        # 2. 리소스 업데이트 및 이력(Audit Log) 남기기
        # 3. 상태를 APPLIED로 업데이트
        await asyncio.sleep(60)
```

## 5. 성과 및 학습 포인트 (Outcome & Learning)
- **운영 자동화**: 날짜를 정해두면 서버가 알아서 업데이트하므로 관리자의 개입이 줄어들었습니다.
- **안정성 확보**: 비동기 루프 내에서의 DB 세션 관리(Scoped Session) 및 예외 처리 기법을 정립했습니다.

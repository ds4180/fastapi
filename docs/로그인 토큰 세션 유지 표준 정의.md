# 🔐 로그인, 토큰 및 세션 유지 표준 정의 (Authentication & Session Management)

본 문서는 프로젝트의 보안 핵심인 JWT 토큰 기반 인증과 SvelteKit 서버 사이드 훅(Hooks)을 이용한 세션 유지 매커니즘을 상세히 기술합니다. **이 로직은 보안 및 사용자 경험(UX)과 직결되므로 수정 시 매우 주의해야 합니다.**

---

## 1. 인증 아키텍처 개요 (Overview)

본 시스템은 **"자바스크립트가 접근할 수 없는 보안 환경"**을 구축하기 위해 `httpOnly Cookie` 방식을 채택하고 있습니다.

- **Storage**: `access_token`, `refresh_token`, `username`을 브라우저의 `localStorage`가 아닌 서버 사이드 쿠키(`httpOnly`)에 저장합니다.
- **Security**: XSS(Cross-Site Scripting) 공격으로부터 토큰 탈취를 원천 차단합니다.
- **Session Layer**: SvelteKit의 `hooks.server.js`가 모든 API 요청의 중재자(Proxy) 역할을 수행합니다.

---

## 2. 핵심 컴포넌트 상세 설명

### 2.1. SvelteKit `hooks.server.js` (The Heart)

`handleFetch` 함수는 서버 사이드(`load`, `actions`)에서 발생하는 모든 백엔드 API 요청을 가로칩니다.

#### 🛠️ 자동 인증 헤더 주입 (Auto Injection)
서버 사이드에서 `fetch`가 발생할 때, 쿠키에 저장된 `access_token`을 읽어 백엔드가 요구하는 `Authorization: Bearer <token>` 헤더를 자동으로 추가합니다.

#### 🔄 무중단 세션 유지 (Multi-Slot 지원)
이제 시스템은 기기 카테고리별로 **최대 2개의 세션 슬롯**을 허용합니다. 따라서 동일 사용자가 PC 탭을 여러 개 열거나 폰에서 동시에 접속해도 로그인이 풀리지 않습니다.

---

## 3. 주의사항 및 유지보수 가이드 (CRITICAL)

### ⚠️ `errorData` 참조 오류 주의
`hooks.server.js` 내부에서 응답 본문을 파싱할 때, `let errorData;` 선언이 주석 뒤에 숨거나 누락되지 않도록 주의해야 합니다.

### ⚠️ 응답 클론(Clone) 필수
`response.json()`은 스트림을 소비하므로, 본문을 읽은 후 원래 요청을 재시도하거나 반환해야 할 경우 반드시 `response.clone()`을 사용해야 합니다.

### ⚠️ 리다이렉트 가로채기 금지 (IMPORTANT)
SvelteKit의 `redirect()` 함수는 특수 에러 객체를 `throw`하여 작동합니다. 따라서 `load` 함수 등에서 `try...catch`를 사용할 경우, **리다이렉트 에러를 일반 에러로 오해하여 삼켜버리지 않도록** 다음과 같이 처리해야 합니다.
```javascript
try {
    // ... fetch 로직
} catch (e) {
    // 💡 리다이렉트(302, 303)는 catch에서 다시 던져야 브라우저가 이동함
    if (e.status === 302 || e.status === 303) throw e;
    // 그 외 진짜 에러 처리
}
```

## 4. 데이터 흐름 (Sequence)

1. **Client**: 페이지 이동 또는 데이터 요청.
2. **Svelte Server (Hooks)**: 쿠키 확인 -> 헤더 추가 -> 백엔드 요청.
3. **Back-end**: Redis Set에서 JTI 포함 여부 확인 -> (불일치 시) 401 응답 -> (정상 시) 데이터 반환.
4. **Svelte Server (Hooks)**: 
    - 401 발생 시: 중복 로그인(Kicked) 메시지 확인 후 쿠키 삭제 및 리다이렉트.
5. **Client**: 최종 결과 수신 및 화면 렌더링.

---

## 5. 클라이언트 사이드 API 호출 제한 (CRITICAL)

본 프로젝트는 보안을 위해 `httpOnly` 쿠키를 사용하므로, 브라우저의 자바스크립트는 토큰에 직접 접근할 수 없습니다. 따라서 다음과 같은 원칙을 반드시 준수해야 합니다.

### 🚫 클라이언트 측 직접 fetch 금지
- 브라우저 단(`+page.svelte` 내 script 등)에서 `fetch('/api/...')`를 직접 호출하면 `Authorization` 헤더가 누락되어 **401 Unauthorized** 오류가 발생합니다.

### ✅ 서버 사이드 Proxy 호출 권장
- 모든 인증이 필요한 API 요청은 SvelteKit의 **`+page.server.js`**에서 수행해야 합니다.
- 서버 사이드에서 실행되는 `fetch`는 자동으로 인증 헤더가 주입됩니다.
- 데이터 갱신 시에는 `invalidateAll()`을 사용하여 서버 로직을 재호출하십시오.

# 🔐 로그인, 토큰 및 세션 유지 표준 정의 (Authentication & Session Management)

본 문서는 프로젝트의 보안 핵심인 JWT 토큰 기반 인증과 SvelteKit 서버 사이드 훅(Hooks)을 이용한 세션 유지 매커니즘을 상세히 기술합니다. **이 로직은 보안 및 사용자 경험(UX)과 직결되므로 수정 시 매우 주의해야 합니다.**

---

## 1. 인증 아키텍처 개요 (Overview)

본 시스템은 **"자바스크립트가 접근할 수 없는 보안 환경"**을 구축하기 위해 `httpOnly Cookie` 방식을 채택하고 있습니다.

- **Storage**: `access_token`, `refresh_token`, `username`을 브라우저의 `localStorage`가 아닌 서버 사이드 쿠키(`httpOnly`)에 저장합니다.
- **Security**: XSS(Cross-Site Scripting) 공격으로부터 토큰 탈취를 원천 차단합니다.
- **Session Layer**: SvelteKit의 `hooks.server.js`가 모든 API 요청의 중재자(Proxy) 역할을 수행합니다.

---

## 2. 핵심 컴포넌트 상세 설명

### 2.1. SvelteKit `hooks.server.js` (The Heart)

`handleFetch` 함수는 서버 사이드(`load`, `actions`)에서 발생하는 모든 백엔드 API 요청을 가로챕니다.

#### 🛠️ 자동 인증 헤더 주입 (Auto Injection)
서버 사이드에서 `fetch`가 발생할 때, 쿠키에 저장된 `access_token`을 읽어 백엔드가 요구하는 `Authorization: Bearer <token>` 헤더를 자동으로 추가합니다.
```javascript
const access_token = event.cookies.get('access_token');
if (access_token && request.url.startsWith(env.SERVER_URL)) {
    request.headers.set('Authorization', `Bearer ${access_token}`);
}
```

#### 🔄 무중단 세션 유지 (Silent Refresh)
만약 `access_token`이 만료되어 백엔드에서 `401 Unauthorized` 응답이 오면, 사용자가 알지 못하게 다음 과정을 수행합니다:
1. 쿠키에서 `refresh_token`을 꺼내 백엔드의 `/users/refresh` API 호출.
2. 새 토큰 발급 성공 시, 쿠키를 갱신하고 **실패했던 원래의 요청을 새 토큰으로 재시도**.
3. 이를 통해 사용자는 로그아웃 없이 작업을 이어갈 수 있습니다.

#### 🚫 중복 로그인 방지 (Kicked Out)
백엔드 응답이 `401`이면서 본문에 `"다른 기기에서 로그인하여 접속이 종료되었습니다."`라는 메시지가 포함된 경우:
- 즉시 모든 쿠키를 삭제하고 로그인 페이지로 리디렉션합니다.
- URL 파라미터로 `?reason=kicked`를 넘겨 클라이언트에서 알림을 띄우게 합니다.

---

### 2.2. 백엔드 (FastAPI) 세션 관리

- **`UserSession` 테이블**: 현재 활성화된 기기 정보(IP, Device Type, Session Key)를 저장합니다.
- **Redis 연동**: 실시간 세션 검증 및 강제 로그아웃(Kick) 기능을 위해 Redis를 활용합니다.
- **Token Payload**: `sub`(사용자ID), `jti`(세션고유ID), `category`(기기분류)를 포함하여 정밀한 검증을 수행합니다.

## 3. 주의사항 및 유지보수 가이드 (CRITICAL)

### ⚠️ `errorData` 참조 오류 주의
`hooks.server.js` 내부에서 응답 본문을 파싱할 때, `let errorData;` 선언이 주석 뒤에 숨거나 누락되면 시스템 전체가 **500 Internal Server Error**를 내며 마비됩니다. 
- **원인**: `response.json()`을 호출하기 전 변수 선언이 주석 처리됨.
- **해결**: 주석과 코드를 엄격히 분리하여 선언문이 확실히 실행되도록 유지해야 합니다.

### ⚠️ 응답 클론(Clone) 필수
`response.json()`은 스트림을 소비하므로, 본문을 읽은 후 원래 요청을 재시도하거나 반환해야 할 경우 반드시 `response.clone()`을 사용해야 합니다.

---

## 4. 데이터 흐름 (Sequence)

1. **Client**: 페이지 이동 또는 데이터 요청.
2. **Svelte Server (Hooks)**: 쿠키 확인 -> 헤더 추가 -> 백엔드 요청.
3. **Back-end**: 토큰 검증 -> (만료 시) 401 응답 -> (정상 시) 데이터 반환.
4. **Svelte Server (Hooks)**: 
    - 401 발생 시: 토큰 재발급 시도 -> 성공 시 재요청 / 실패 시 로그인 이동.
    - Kicked 메시지 발견 시: 쿠키 삭제 -> 로그인 강제 이동.
5. **Client**: 최종 결과 수신 및 화면 렌더링.

# 🔐 로그인, 토큰 및 세션 유지 표준 정의 (Authentication & Session Management) - v1.8

이 문서는 수많은 시행착오와 **"잘 되던 코드가 보안 강화(httpOnly) 후 먹통이 되는 고통"**을 겪으며 정립된 본 프로젝트의 성전(Bible)입니다. 미래의 개발자와 AI는 이 표준을 어겨 동일한 실패를 반복하지 마십시오.

---

## 0. [회고] 실패의 역사와 해결책 (War Story)

### ❌ 과거의 실패 과정 (반복된 삽질의 기록)
1.  **초기 단계**: `api.js`에서 자바스크립트로 토큰을 다룸. 코딩은 쉬웠으나 보안이 취약했음.
2.  **보안 강화**: `httpOnly` 쿠키를 적용하자마자 잘 되던 `api.js` 기반 코드가 모두 작동 중지됨 (JS가 쿠키를 읽지 못함).
3.  **혼란의 시기**: AI들이 "표준 fetch를 써라", "헤더에 수동으로 넣어라" 등 잘못된 대안을 제시하여 수차례 헛수고를 반복함.
4.  **최종 해결**: 모든 인증 로직을 **`hooks.server.js` (서버 단)**로 옮기고, SvelteKit 전용 `event.fetch`를 사용하면서 비로소 안정화됨.

### 💡 왜 이 방식(Hooks)이 유일한 정답인가?
- **httpOnly의 벽**: 브라우저의 JS는 토큰을 만질 수 없습니다. 오직 서버(`Hooks`)만이 쿠키를 열고 새 토큰을 수리할 수 있습니다.
- **SSR의 특성**: 서버 사이드에서 데이터를 가져올 때, 표준 `fetch`는 세션 정보를 잃어버립니다. 오직 `event.fetch`만이 사용자의 신분을 백엔드에 증명할 수 있습니다.

## 1. 🛡️ 이중 방어 시스템 (The Dual-Gate System)

본 시스템은 **"입구에서 막고, 도중에도 잡는다"**는 이중 방어 전략을 채택합니다.

### 1.1. [관문] 선제적 갱신 (Proactive: `handle` Hook)
- **위치**: `hooks.server.js` 내 `handle` 함수.
- **역할**: 사용자가 페이지를 열기 **직전**에 쿠키 상태를 확인.
- **핵심**: 액세스 토큰이 만료되었다면 `load` 함수들이 실행되기 전에 미리 토큰을 갱신함. 이 로직이 없으면 특정 페이지 진입 시 "귀신같이" 로그아웃되는 현상을 막을 수 없음.

### 1.2. [안전장치] 사후 재시도 (Reactive: `handleFetch` Hook)
- **위치**: `hooks.server.js` 내 `handleFetch` 함수.
- **역할**: 데이터 통신 **도중**에 토큰이 만료되어 401 에러가 났을 때 작동.
- **핵심**: 요청을 복제(Clone)하여 보관했다가, 토큰 갱신 성공 즉시 새 토큰으로 재시도하여 사용자에게 에러를 노출하지 않음.

## 2. ⚠️ 미래를 위한 금기 사항 (CRITICAL WARNING)

**다음 사항을 위반하면 시스템 전체의 인증이 붕괴됩니다.**

1.  **`api.js` 위주의 수정 금지**: 브라우저 단에서 토큰을 직접 가공하거나 헤더를 주입하려 하지 마십시오.
2.  **표준 `fetch()` 사용 금지**: 서버 사이드 로직에서는 반드시 SvelteKit이 제공하는 래퍼 `fetch`를 사용하십시오.
3.  **리다이렉트 에러 가로채기 금지**: `try...catch` 사용 시 SvelteKit의 `redirect` 에러(302, 303)를 삼키지 말고 다시 `throw` 하십시오.
4.  **AI 보조 도구 필독**: 사용자에게 `httpOnly` 환경에서 클라이언트 사이드 인증 수정을 제안하는 것은 시간 낭비일 뿐입니다. 모든 정답은 `hooks.server.js`에 있습니다.

# 🔐 다중 접속 및 로그인 세션 관리 표준

이 문서는 Redis를 활용한 사용자 세션 관리 및 다중 접속 제한 정책에 대해 정의합니다. 본 시스템은 보안 강화와 데이터 무결성 보장을 위해 "기기별 단일 세션 유지" 정책을 채택하고 있습니다.

---

## 1. 핵심 정책: 기기별 단일 세션 (Kicking Out)
* **목적**: 계정 공유 방지 및 분실 기기에서의 무단 접속 차단.
* **정책**: 동일한 사용자 아이디로 새로운 기기(또는 브라우저)에서 로그인할 경우, **기존에 접속해 있던 동일 카테고리 기기의 세션은 즉시 무효화**됩니다.
* **기기 카테고리 (Device Category)**:
    * **WORKSPACE**: PC, 노트북 등 사무 환경.
    * **MOBILE**: 스마트폰, 태블릿 등 모바일 환경.
    * *참고: WORKSPACE와 MOBILE은 서로 별개로 관리되므로, PC와 폰에서 동시에 한 명의 유저가 접속하는 것은 허용됩니다.*

## 2. 기술적 구현 메커니즘 (Redis & JTI)

### ① 고유 세션 키 (JTI)
* 로그인 시마다 UUID 기반의 고유한 **JTI(JWT ID)**를 생성하여 토큰에 심습니다.
* 이 JTI는 해당 로그인의 '고유 입장권 번호' 역할을 합니다.

### ② Redis 슬롯 관리
* Redis에 `session:{user_id}:{category}` 형식의 키로 최신 발송된 JTI를 저장합니다.
* **만료 시간**: 7일 (사용자가 일주일간 활동하지 않으면 자동 로그아웃).

### ③ 2중 검증 프로세스
모든 API 요청 및 토큰 재발급(Refresh) 시 다음 과정을 거칩니다.
1. **토큰 유효성 검사**: JWT 서명 및 만료 시간 확인.
2. **Redis 세션 검사**: 토큰에 담긴 JTI가 Redis에 저장된 '최신 JTI'와 일치하는지 확인.
   - 일치하지 않으면 "다른 기기에서 로그인되었습니다"라는 메시지와 함께 401 에러를 반환하고 즉시 로그아웃 처리합니다.

## 3. 토큰 및 쿠키 수명 설정

| 구분 | 기간 | 역할 |
| :--- | :--- | :--- |
| **Access Token** | 1일 | 실제 API 호출에 사용되는 단기 열쇠 |
| **Refresh Token** | 7일 | Access Token 만료 시 자동으로 연장하는 장기 열쇠 |
| **SvelteKit Cookie** | 1일 | 브라우저에 저장되는 토큰의 실제 보관 수명 |

## 4. 세션 종료 유형 및 상태값 (Status)

시스템은 세션이 종료된 원인을 다음과 같이 구분하여 기록합니다.

| 상태값 | 의미 | 발생 조건 | Redis 처리 |
| :--- | :--- | :--- | :--- |
| **ACTIVE** | 현재 접속 중 | 정상 로그인 완료 시 | JTI 저장 및 유지 |
| **LOGOUT** | 직접 로그아웃 | 사용자가 [로그아웃] 버튼을 클릭함 | **즉시 삭제** |
| **KICKED_OUT** | 중복 접속 밀어냄 | 동일 카테고리에서 다른 기기 로그인 발생 | **새 JTI로 덮어씀** |

## 5. 운영 및 관리
* **세션 모니터링**: 관리자 페이지에서 현재 활성화된 세션 목록을 실시간으로 확인할 수 있습니다.
* **강제 로그아웃**: 관리자는 특정 세션을 Redis에서 삭제하여 즉시 해당 유저를 쫓아낼 수 있습니다.

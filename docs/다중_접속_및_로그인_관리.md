# 🛡️ 지능형 다중 접속 및 로그인 슬롯 관리 시스템 (v1.0)

이 문서는 프로젝트의 보안성과 편의성을 동시에 극대화하기 위해 구축된 **'Redis 기반 2슬롯 세션 관리 시스템'**의 정의, 설계 및 구현 상세를 기술합니다.

---

## 1. 시스템 정의 및 목적 (The Concept)

### ① 지능형 2슬롯 (Intelligent Slots)
*   **정의**: 한 사용자당 **MOBILE(폰)** 1대, **WORKSPACE(PC/패드)** 1대의 동시 접속을 허용합니다.
*   **목적**: "내 폰"으로는 항상 알림을 수신하고, "작업용 기기"는 내가 현재 손에 든 기기만 활성화하여 업무 효율과 계정 보안을 동시에 잡습니다.

### ② 선입선출 밀어내기 (FIFO Kicking)
*   **정의**: 같은 카테고리(예: WORKSPACE)에 새로운 기기가 로그인하면, 기존 기기를 즉시 로그아웃(Kick Out) 시킵니다.
*   **목적**: 로그아웃 버튼을 누르지 않아도 기기 이동 시 세션이 자동 정리되게 하여 사용자 경험(UX)을 개선합니다.

---

## 2. 핵심 아키텍처 (Architecture)

### ① Redis: Hot Session Storage (고성능 실시간 검증)
*   **역할**: "현재 이 유저의 이 슬롯 주인은 누구인가?"를 저장합니다.
*   **키 구조**: `session:{user_id}:{device_category}` → `Value: JWT_JTI`
*   **장점**: 메모리에서 작동하여 API 요청마다 발생하는 검증 부하를 최소화하며, TTL(7일)을 통해 좀비 세션을 자동 제거합니다.

### ② DB (PostgreSQL): Session Log (6개월 이력 추적)
*   **역할**: 모든 로그인, 로그아웃, 밀려남(`KICKED_OUT`) 기록을 보관합니다.
*   **장점**: 사고 발생 시 "누가, 언제, 어떤 기기"로 접속했는지 증거를 남기며, 운영 통계 자료로 활용합니다.

---

## 3. 구현 상세 (Implementation)

### ① 백엔드 (FastAPI)
*   **`UserSession` 모델**: `status`(ACTIVE, KICKED_OUT), `session_key`(JTI) 등을 포함한 로그 테이블.
*   **`login_for_access_token`**: 
    1. 로그인 시 `device_category` 수신.
    2. 고유 식별자 `jti`(UUID) 생성 및 토큰 삽입.
    3. Redis에 `jti` 저장 (기존 값 덮어쓰기로 자동 밀어내기).
    4. DB에 이전 세션을 `KICKED_OUT`으로 업데이트 후 새 로그 생성.
*   **`get_current_user`**: 매 요청 시 토큰의 `jti`와 Redis의 `jti`를 비교. 불일치 시 `401 Unauthorized` (메시지: "다른 기기에서 로그인하여...") 반환.

### ② 프론트엔드 (SvelteKit)
*   **`+page.server.js`**: 서버 사이드에서 `User-Agent`를 분석해 `MOBILE` / `WORKSPACE`를 판별하여 백엔드에 전달.
*   **`hooks.server.js`**: 전역 가로채기 도구. 백엔드에서 401 에러(밀려남)를 받으면 즉시 쿠키를 삭제하고 `/user-login?reason=kicked`로 강제 압송.
*   **`/login-manage` 페이지**: 관리자가 실시간 접속 현황을 모니터링하고 수동으로 '강제 종료'를 실행하는 관제 센터.

## 4. 실행 및 운영 방법 (Operation Guide)

### ① 환경 구축 (Redis 필요)
`docker-compose.yml`에 Redis 서비스가 반드시 포함되어야 합니다.
```yaml
redis:
  image: redis:latest
  command: redis-server --requirepass "1234"
```

### ② 주소 체계 표준 (The Slash Standard)
*   모든 API 호출은 `/api/` 접두사를 사용하지만, 백엔드 내부 라우터와 Svelte 서버 내부 `fetch` 시에는 `/api`를 제외한 주소를 사용합니다. (Nginx 프록시 설정 준수)

### ③ 로그 관리 정책
*   `UserSession` 데이터는 인덱싱 처리되어 있으며, 6개월이 지난 데이터는 `cleanup_gc.py` 스케줄러에 의해 자동으로 삭제되도록 설계되었습니다.

## 5. 엔지니어링 노트 (Strict Feedback Notes)

*   **보안**: `httpOnly` 쿠키를 사용하여 자바스크립트에 의한 토큰 탈취를 원천 차단했습니다.
*   **성능**: `get_current_user` 시 DB 조회보다 Redis 조회를 우선하여 응답 속도를 챙겼습니다.
*   **UX**: 튕겨나간 사용자에게 단순 에러가 아닌 명확한 사유("다른 기기 로그인")를 알림창으로 안내합니다.

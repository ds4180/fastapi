# 🔐 다중 접속 및 로그인 세션 관리 표준 (v1.1)

이 문서는 Redis를 활용한 사용자 세션 관리 및 다중 접속 제한 정책에 대해 정의합니다. 본 시스템은 보안성과 사용자 편의성(UX) 사이의 균형을 위해 **"기기 카테고리별 다중 세션 슬롯"** 정책을 채택하고 있습니다.

---

## 1. 핵심 정책: 기기 카테고리별 다중 슬롯 (Multi-Session Slot)
* **목적**: 탭 여러 개 사용 시의 튕김 방지 및 불안정한 네트워크 환경에서의 로그인 유지 안정성 확보.
* **정책**: 동일한 사용자 아이디로 동일 카테고리 기기에서 **최대 2개까지의 동시 접속**을 허용합니다.
* **슬롯 관리**: 3번째 로그인이 발생하면, 가장 오래된 세션 하나가 자동으로 밀려납니다 (FIFO 또는 Random Eviction).
* **기기 카테고리 (Device Category)**:
    * **WORKSPACE**: PC, 노트북 등 브라우저 환경.
    * **MOBILE**: 스마트폰, 태블릿 등 앱/모바일 브라우저 환경.
    * *참고: WORKSPACE와 MOBILE은 별도의 슬롯(각 2개씩)을 가집니다.*

## 2. 기술적 구현 메커니즘 (Redis Set & JTI)

### ① 고유 세션 키 (JTI)
* 로그인 시마다 UUID 기반의 고유한 **JTI(JWT ID)**를 생성하여 토큰에 심습니다.
* 이 JTI는 개별 세션의 '고유 입장권 번호'입니다.

### ② Redis 집합(Set) 구조 사용
* Redis 키: `session:{user_id}:{category}`
* 데이터 구조: **Set (집합)**
* **작동 방식**: 새로운 로그인이 발생할 때마다 해당 집합에 `SADD` 명령으로 JTI를 추가합니다.
* **슬롯 제어**: `SCARD` 명령으로 개수를 확인하고, 허용 범위(2개) 초과 시 기존 JTI를 제거합니다.

### ③ 포함 관계 검증 (Membership Check)
모든 API 요청 시 백엔드(`get_current_user`)는 다음 과정을 거칩니다.
1. **토큰 분석**: 토큰에서 JTI와 카테고리를 추출합니다.
2. **집합 멤버십 확인**: `SISMEMBER` 명령을 사용하여 **현재 JTI가 Redis의 해당 세션 집합에 포함되어 있는지** 확인합니다.
   - 포함되어 있지 않다면 "다른 기기에서 로그인되었습니다"라는 메시지와 함께 401 에러를 반환합니다.

## 3. 토큰 및 쿠키 수명 설정

| 구분 | 기간 | 역할 | 비고 |
| :--- | :--- | :--- | :--- |
| **Access Token** | 1일 | 실제 API 호출용 | 넉넉한 수명으로 사용자 편의 증대 |
| **Session Key (Redis)** | 7일 | 실시간 유효성 검증 | 활동 중단 시 7일 후 자동 소멸 |
| **httpOnly Cookie** | 1일 | 브라우저 저장용 | JS 접근 불가 (XSS 방어) |

## 4. 세션 상태 및 관리

| 상태값 | 의미 | Redis 처리 |
| :--- | :--- | :--- |
| **ACTIVE** | 현재 유효한 세션 | Redis Set 내에 JTI 존재 |
| **LOGOUT** | 직접 로그아웃 | `SREM` 명령으로 해당 JTI 즉시 제거 |
| **KICKED_OUT** | 슬롯 초과로 밀려남 | 새 로그인이 발생하여 기존 JTI가 Set에서 빠짐 |

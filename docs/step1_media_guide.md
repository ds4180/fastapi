# Step 1: 미디어 시스템 고도화 가이드

이 문서는 게시판 프로젝트의 1단계인 이미지 및 문서 업로드 시스템의 구현 내용과 주요 설계 결정을 정리합니다.

## 1. 개요
단순한 파일 저장을 넘어, 재사용 가능하고 안정적인 미디어 처리 아키텍처를 구축하는 것을 목표로 합니다.

## 2. 주요 아키텍처

### 백엔드 (FastAPI)
- **`fileupload_service.py`**: 비즈니스 로직 전담. 파일 저장, 썸네일 생성, 이름 변경(격리) 담당.
- **`fileupload_router.py`**: 독립적인 업로드 API 엔드포인트 제공.
- **`question_crud.py`**: 게시글과 이미지 간의 1:N 관계 관리 및 데이터 정합성 유지.

### 프론트엔드 (SvelteKit)
- **`ImageUploader.svelte`**: **자립형(Self-Contained) 컴포넌트.** 파일 선택, 업로드, 결과 목록 렌더링(이미지 썸네일/문서 아이콘), 개별 삭제 기능을 모두 캡슐화. `bind:uploadedFiles`를 통해 상위 페이지와 실시간 동기화.
- **상세 페이지 (Lightbox)**: 썸네일 위주의 빠른 로딩과 클릭 시 원본 보기 기능.
- **에러 핸들링**: `Error.svelte` 컴포넌트와의 연동을 통해 서버 측 검증 결과를 즉시 UI에 반영.

---

## 3. 핵심 기능 상세

### 📄 파일 형식 제한 및 통합 업로드
- **정책**: '이미지'와 '문서'(PDF, DOC, XLS 등)만 허용하며, **동영상은 제외**합니다.
- **UI**: '이미지 업로드'에서 '업로드'로 레이블 범용화.
- **표시**: 이미지는 썸네일로, 문서는 아이콘으로 미리보기 제공.

### 📸 서버 사이드 썸네일 자동 생성 (이미지 한정)
- **기술**: Python `Pillow` 라이브러리 활용.
- **정책**: 900x900 사이즈를 기준으로 이미지 업로드 시 무조건 생성.
- **효과**: 리스트 및 수정 페이지 로딩 속도 최적화.

### 🛡️ 파일 격리 시스템 (Deleted Prefix)
- **개념**: 파일 삭제 시 실제 하드 삭제 대신 파일명 앞에 `deleted_` 접두어를 붙여 격리.
- **수정 로직**: 질문 수정 시, 기존 이미지 중 새 리스트에서 빠진 파일만 정확히 골라내어 `deleted_` 처리.
- **안정성**: 실수로 인한 데이터 유실 방지 및 추후 일괄 청소(GC) 가능 구조.

### ⛓️ 데이터 정합성 (Session Consistency)
- **문제**: 벌크 삭제(`delete()`) 시 DB와 메모리 간 불일치로 500 에러 발생.
- **해결**: SQLAlchemy의 `cascade="all, delete-orphan"` 설정을 활용하여 `images.clear()` 방식으로 객체 관계를 직접 관리.

# 전역 알림 시스템 표준 정의 (Notification System Standards)

이 문서는 프로젝트 전체에서 공통으로 사용될 알림 및 업무 지시 시스템의 표준 스펙을 정의합니다. 본 시스템은 단순한 UI 요소가 아닌, 사내 업무 이행 보장 및 책임 소지 명확화를 위한 핵심 도구입니다.

## 1. 시스템 목적
* **업무 지시의 확실한 전달**: 밴드(Band) 등 단순 게시판의 한계를 넘어, 중요 공지를 반드시 정독하게 함.
* **책임 소지 방어**: "확인하지 못했다"는 변명을 시스템적으로 차단하여 관리적 과실을 방지.
* **개발 효율성**: 표준화된 호출 방식을 통해 어떤 페이지에서든 디자인 고민 없이 즉시 알림 송출.

## 2. 알림 등급 및 특종 동작 (Levels)

| 등급 | 명칭 | 형태 | 동작 특징 | 주요 용도 |
| :--- | :--- | :--- | :--- | :--- |
| **Level 1** | **일반 알림** | Toast (팝업) | 3초 후 자동 소멸, 사용자 클릭 방해 없음 | 저장됨, 삭제 완료, 가벼운 안내 |
| **Level 2** | **상태 배너** | Fixed Banner | 해결 전까지 상/하단 고정 | 네트워크 불안정, 서버 점검 예고 |
| **Level 3** | **일반 확인** | Modal (중앙) | 사용자의 확인 클릭 전까지 화면 차단 | 삭제 재확인, 규정 변경 인지 |
| **Level 4** | **강제 지시** | Timed Modal | **지정 시간 대기 필수 + 클릭 시 타이머 리셋** | 배차 공지, 긴급 안전 지시 |
| **Level 5** | **시스템 락** | Full Screen | 내비게이션 포함 전체 기능 차단 | 비인가 유저, 시스템 전면 긴급점검 |

### 🛠 Level 4 "인과응보" 로직
* **목적**: 정독 유도.
* **작동**: 확인 버튼에 5~10초의 카운트다운 표시.
* **패널티**: 시간이 다 되기 전에 버튼을 클릭할 경우 **타이머가 최초 설정 시간으로 리셋**되며 얄미운 경고 문구가 출력됨 (예: "지시 사항을 정독하십시오.")

## 3. 알림 범주 및 노출 경로 제어 (Routing)

알림 설정 시 '노출 경로' 필드를 통해 알림이 나타날 범위를 정밀하게 제어합니다.

1. **전역 알림 (Global)**
   - **설정**: 노출 경로 필드를 **비워둠 (Empty)**.
   - **동작**: 사용자가 어떤 메뉴에 있든 즉시 나타남.
2. **대상 지정 알림 (Targeted)**
   - **설정**: 특정 페이지 주소를 입력 (예: `/question-list`).
   - **다중 지정**: 콤마(`,`)로 구분하여 여러 페이지 지정 가능 (예: `/question-list, /question-create`).
   - **동작**: 사용자가 해당 경로에 접속했을 때만 알림이 발생.
3. **액션 피드백 (Action)**
   - **설정**: 코드 내에서 직접 호출.
   - **동작**: 버튼 클릭 등 특정 액션 직후에 일회성으로 발생.

## 4. 표준 데이터 규격 (Interface)

모든 알림은 아래의 데이터 형식을 따르며, 전역 스토어(`notify`)를 통해 송출함.

```javascript
{
  id: "unique_id",          // 중복 방지 및 확인 기록용
  type: "toast" | "modal",   // UI 형태
  style: "info" | "success" | "warning" | "danger", // 시각적 테마
  message: "공지 내용",      // 본문
  level: 1 | 2 | 3 | 4 | 5,  // 작동 등급
  resetSec: 5,               // Level 4용 대기 시간
  confirmText: "확인함",      // 버튼 텍스트
  route: "/question-list",   // 노출 경로 (비어있으면 전역, 콤마로 다중지정 가능)
  redirect_url: "/news/123", // (선택) 확인 클릭 시 강제 이동할 주소
  dontShowDay: 1             // (선택) 오늘 하루 안보기 사용 여부
}
```

## 5. DB 스키마 상세 (Back-end)

알림 관리 및 송출을 위한 `Alert` 테이블 스펙입니다.

| 필드명 | 타입 | 설명 |
| :--- | :--- | :--- |
| **id** | Integer | 고유 식별자 (PK) |
| **message** | Text | 알림 본문 내용 |
| **level** | Integer | 1~5 (등급 체계 참조) |
| **style** | String | info, success, warning, danger |
| **route** | String | 노출 조건 경로 (비어있으면 전역 노출) |
| **redirect_url** | String | 확인 클릭 시 강제 이동할 대상 경로 |
| **is_active** | Boolean | **적용/중지** (관리자의 수동 제어) |
| **start_date** | DateTime | **예약 시작** (노출 개시 시점) |
| **end_date** | DateTime | **노출 종료** (자동 소멸 시점) |
| **target_users** | Text | **대상 필터** (근무/휴무 상태 등 필터링 조건) |
| **reset_sec** | Integer | Level 4 전용 강제 대기 초 설정 |
| **create_date** | DateTime | 생성 일시 |

## 6. 관리 및 구현 가이드라인
* **알림-본문 연동 (SOP)**: 알림은 "중요 내용이 게시됨"을 알리는 입구(Trigger) 역할에 집중하며, 확인 클릭 시 `redirect_url`을 통해 해당 공지 본문 페이지로 강제 이동시켜 '읽음' 기록을 유도함.
* **알림 관리 (CRUD)**: 전용 관리 페이지에서 전체 공지 리스트를 관리하며, 시작/종료일 및 노출 경로를 제어함.
* **UI 격리 (The Agent)**: 별도의 `AlertManager` 컴포넌트가 `+layout.svelte`에서 상주하며 전역 스토어 및 현재 경로(`$page.url.pathname`)를 감시하여 알림을 노출함.
* **최종 보루 (SOP)**: 시스템 장애(365일 중 5일)를 대비하여, 중요 업무 안내는 아날로그(종이 출력) 이력 관리와 병행함.

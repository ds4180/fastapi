# 🌐 실시간 웹소켓(WebSocket) 도입 및 시스템 고도화 기록

이 문서는 프로젝트 내 실시간 통신 기능을 구현하기 위해 웹소켓을 도입하고, 기존 시스템의 통신 구조를 표준화한 과정을 기록한 문서입니다.

## 📅 작업 일자: 2026-02-26

## 🚀 1. 배경 및 문제점 분석

### ① 통행권(/api) 중복 및 404 에러
*   **현상:** 프론트엔드에서 `/api/alert` 호출 시 작동하지 않고, `/api/api/alert`로 호출해야만 통신이 되는 문제 발생.
*   **원인:** Nginx 설정(`proxy_pass`) 끝에 슬래시(`/`)가 있어 입구에서 `/api`를 잘라내는데, 백엔드 라우터(FastAPI)가 내부에서 `/api`를 한 번 더 요구하고 있었음 (이중 잠금 상태).

### ② 실시간성 부재 (30초 폴링의 한계)
*   **현상:** 관리자가 알림을 등록해도 기사님들이 최대 30초를 기다려야 화면에 나타남.
*   **목표:** 등록 즉시 0.1초 만에 화면에 나타나는 "진정한 실시간" 구현 필요.

### ③ 신호 유실 문제 (동기 함수의 한계)
*   **현상:** 웹소켓 기능을 처음 넣었을 때, 메시지는 저장되지만 실시간 신호가 전달되지 않음.
*   **원인:** 일반 동기 함수(`def`) 내에서 비동기 신호(`broadcast`)를 쏘려다 보니 엔진 불일치로 신호가 씹힘.

## 🛠️ 2. 해결 및 구현 과정

### 단계 1: 주소 체계 표준화 (The Slash Standard)
*   **백엔드:** 모든 라우터(`prefix`)에서 `/api`를 제거하여 Nginx가 잘라준 알맹이 주소만 받도록 수정.
*   **서버 측 페치:** SvelteKit 서버(`+page.server.js`)가 백엔드(8000포트)에 직접 쏠 때 `/api` 없이 호출하도록 일괄 변경.
*   **클라이언트:** 브라우저에서는 항상 `/api/`를 붙여 Nginx에게 API 요청임을 알리도록 통일.

### 단계 2: 범용 웹소켓 도메인(`domain/ws`) 구축
*   특정 기능에 종속되지 않도록 별도의 웹소켓 관리 서비스 구축.
*   **핵심 로직:**
    *   **Ping-Pong:** 60초마다 서버가 핑을 쏘고 클라이언트가 응답하여 연결 유지(Keep-alive).
    *   **Parallel Broadcast:** `asyncio.gather`를 사용해 모든 접속자에게 동시에 신호 전송.
    *   **Safe Cleanup:** 전송 실패 시 끊긴 연결을 즉시 감지하고 목록에서 제거.

### 단계 3: 백엔드 완전 비동기(Async)화
*   **변경 전:** `def create_alert` (동기) + `asyncio.create_task` (예약 시도)
*   **변경 후:** `async def create_alert` (비동기) + `await manager.broadcast` (확실한 발송)
*   라우터부터 CRUD까지 `async/await` 체인을 완성하여 신호 발송의 안정성 확보.

### 단계 4: Nginx 웹소켓 전용 도로 개설
*   호스트 설정 파일(`/home/lee/docker/nginx.docker/default.conf`) 수정.
*   `/ws` 경로에 대해 `Upgrade`, `Connection` 헤더를 추가하여 WebSocket 핸드셰이크 허용.
*   `proxy_read_timeout`을 24시간으로 늘려 장시간 연결 유지 보장.

### 단계 5: Svelte 5 반응성 최적화
*   **문제:** 신호를 받아도 스토어의 데이터 참조값이 같으면 화면이 안 바뀜.
*   **해결:** `alerts.set([...json])` 처럼 **새 배열(New Array)**을 생성하여 할당함으로써 Svelte 5가 변화를 즉시 감지하고 UI를 갱신하도록 수정.

## 🏁 3. 최종 결과 및 효과

1.  **즉각적인 알림:** 관리자가 [전송] 버튼을 누르는 순간 모든 기사님 모바일/PC 화면에 즉시 배너 노출.
2.  **안정적인 연결:** 네트워크가 잠시 끊겨도 프론트엔드의 **지수 백오프(Exponential Backoff)** 로직이 자동으로 재연결 수행.
3.  **깔끔한 코드:** `/api` 중복 문제를 완전히 해결하여 어떤 개발자가 봐도 이해하기 쉬운 표준 구조 완성.
4.  **범용성 확보:** 새로 만든 `/ws` 통로를 통해 향후 채팅, 실시간 관제 등 모든 기능을 추가 비용 없이 구현 가능.

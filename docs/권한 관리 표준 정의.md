# 권한 관리 표준 정의 (Authority Management Standard)

이 문서는 시스템의 보안 유지와 업무 효율성을 위해 사용자 분류, 권한 부여, 그리고 데이터 접근 범위(Scope)를 관리하는 표준 규칙을 정의합니다.

---

## 1. 사용자 분류 (User Classification)

사용자는 크게 **가입 여부**와 **직무 성격**에 따라 다음과 같이 분류됩니다.

### 1.1. 미가입자 (Guest)
- 계정이 없거나 로그인하지 않은 사용자.
- 권한: 공개 게시판 열람, 로그인/회원가입.

### 1.2. 운전원 (Driver)
- 시스템 이용자의 약 90%를 차지하는 현장 인력.
- 별도의 등급(Rank) 구분을 두지 않으며, 모든 운전원은 동일한 기본 권한을 가집니다.
- 주요 권한: 본인 배차 확인, 본인 연차 신청, 게시판 글쓰기.

### 1.3. 비운전원 (Staff / Office)
- 사무실 업무 및 시스템 관리를 담당하는 인력.
- **직급(Rank Level)**에 따라 차등적인 권한을 부여받습니다.
    - **L1 (실무자):** 일반 행정, 게시판 관리, 알림 발송 등 기초 업무.
    - **L2 (관리자):** 운전원 연차 승인, 배차표 확정 등 운영 관리 업무.
    - **L3 (최고 관리자):** 시스템 설정, 사용자 계정 관리, 민감 정보(주민번호 등) 조회.

---

## 2. 데이터 아키텍처: 테이블 분리 표준

보안과 업무 정보 관리를 위해 사용자 정보는 두 개의 테이블로 분리하여 관리합니다. 아래 표에 필요한 모든 관리 항목을 정의해 주십시오.

### 2.1. `users` 테이블 (기초 신상 및 인증)
법적으로 민감하거나 인증에 필수적인 정보를 담는 테이블입니다. (수정 가능성 고려)

| 항목명 (Field)   | 데이터 타입 (Type) | 보안 등급    | 비고 (Description/Constraint)                      |
| :------------ | :------------ | :------- | :----------------------------------------------- |
| `id`          | Integer (PK)  | -        | 자동 생성 고유 번호                                      |
| `username`    | String        | PUBLIC   | 로그인 ID (사무실 부여 사번 등)                                  |
| `password`    | String (Hash) | INTERNAL | 초기 비밀번호 부여 후 변경 권장                              |
| `real_name`   | String        | L3 ONLY  | 실명 (개명 시 수정 가능해야 함) |
| `resident_no` | String (Enc)  | L3 ONLY  | 주민번호 (앞/뒷자리 분리 저장 또는 암호화)              |
| `address`     | String        | L3 ONLY  | 주민등록상 법정 주소                                    |
| `phone`       | String        | L2 ONLY  | 개인 연락처 (휴대폰)                                     |

### 2.2. `user_profiles` 테이블 (업무 확장 및 상세 정보)
드라이버 관리, 배차, 행정 업무에 필요한 방대한 데이터를 담는 테이블입니다.

| 항목명 (Field)    | 데이터 타입 (Type) | 접근 권한   | 비고 (Description/Constraint) |
| :------------- | :------------ | :------ | :-------------------------- |
| `user_id`      | Integer (FK)  | -       | `users.id`와 1:1 관계          |
| `nick_name`    | String        | PUBLIC  | 동명이인 구분용 (예: 김영민A, 김영민B) |
| `user_type`    | String        | PUBLIC  | DRIVER / STAFF 구분           |
| `rank_level`   | Integer       | PUBLIC  | 비운전원 직급 (1~3)               |
| `is_active`    | Boolean       | PUBLIC  | 재직 여부 (True/False)          |
| `employee_no`  | String        | PUBLIC  | 공식 사번                          |
| `joined_date`  | Date          | PUBLIC  | 입사일                         |
| `retired_date` | Date          | L2 ONLY | 퇴사일                         |
| `bank_name`    | String        | L2 ONLY | 급여 은행                       |
| `account_no`   | String        | L2 ONLY | 계좌 번호                       |
| `main_vehicle` | String        | L1 ONLY | 주 배정 차량 번호                  |
| `actual_address`| String        | L2 ONLY | 실거주 주소 (법정 주소와 다를 경우)             |
| `license_no`   | String        | L2 ONLY | 면허 번호                       |
| `license_expiry`| Date          | L1 ONLY | 면허 만료일                      |
| `health_check` | Date          | L1 ONLY | 최근 건강검진일                        |
| `ins_guarantee`| Date          | L1 ONLY | 신원보증보험 만료일                        |
| `admin_memo`   | Text          | L2 ONLY | 관리자용 비공개 메모 |

## 3. 권한 관리 3요소 (Role, Permission, Scope)

현대적인 권한 제어를 위해 다음 세 가지 개념을 결합하여 사용합니다.

### 3.1. Role (누구인가?)
`user_type`과 `rank_level`의 조합으로 결정되는 사용자 역할입니다.
- 예: `ROLE_DRIVER`, `ROLE_STAFF_L1`, `ROLE_STAFF_L2`, `ROLE_ADMIN`.

### 3.2. Permission (무엇을 할 수 있는가?)
시스템 내에서 수행 가능한 구체적인 행동의 단위입니다. (`도메인:행위`)
- 예: `board:write` (글쓰기), `dayoff:approve` (연차 승인), `user:manage` (사용자 관리).

### 3.3. Scope (어디까지 가능한가?)
권한이 미치는 데이터의 경계(범위)입니다.
- **OWN (본인):** 본인과 관련된 데이터만 접근 가능 (대부분의 운전원 권한).
- **GLOBAL (전체):** 시스템의 모든 데이터에 접근 가능 (L2 이상 관리자 권한).

---

## 4. 실무 적용 시나리오 예시

| 상황 | 사용자 | Role | Permission | Scope | 결과 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **연차 내역 조회** | 운전원 A | `DRIVER` | `dayoff:view` | **OWN** | 본인의 연차만 보임 |
| **연차 내역 조회** | 팀장 B | `STAFF_L2` | `dayoff:view` | **GLOBAL** | 전 직원의 연차 목록 보임 |
| **게시물 삭제** | 운전원 A | `DRIVER` | `board:delete` | **OWN** | 내가 쓴 글만 삭제 가능 |
| **게시물 삭제** | 실무자 C | `STAFF_L1` | `board:delete` | **GLOBAL** | 부적절한 모든 글 삭제 가능 |
| **주민번호 조회** | 팀장 B | `STAFF_L2` | `user:view_private` | - | **거부** (L3만 가능) |

---

## 5. 보안 및 이행 지침

1. **최소 권한의 원칙:** 모든 사용자는 업무 수행에 필요한 최소한의 권한과 스코프만 부여받아야 합니다.
2. **명시적 권한 확인:** 코드 내에서 단순히 `is_admin`을 체크하는 대신, `has_permission('dayoff:approve')`와 같이 구체적인 권한을 확인해야 합니다.
3. **주민번호 보호:** `users` 테이블의 `resident_no`는 반드시 양방향 암호화되어야 하며, 로그에 기록되어서는 안 됩니다.

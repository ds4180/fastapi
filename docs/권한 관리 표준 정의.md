# 권한 관리 표준 정의 (Authority Management Standard)

이 문서는 시스템의 보안 유지와 업무 효율성을 위해 사용자 분류, 권한 부여, 그리고 데이터 접근 범위(Scope)를 관리하는 표준 규칙을 정의합니다.

---

## 1. 사용자 분류 및 권한 레벨 (Rank Level)

사용자의 권한은 `rank_level` 이라는 단일 숫자로 정의되며, 숫자가 높을수록 더 높은 시스템 권한을 가집니다. (기존 `user_type`은 직무 구분을 위해 사용될 수 있으나, 권한 제어의 핵심은 `rank_level`입니다.)

| 권한 레벨 | 명칭 (Role) | 주요 대상 | 데이터 접근 범위 (Scope) |
| :--- | :--- | :--- | :--- |
| **0** | `ROLE_DRIVER` | 운전원 (현장 인력) | **OWN** (본인 데이터 위주) |
| **1** | `ROLE_STAFF_L1` | 사무직 실무자 | **GLOBAL** (일반 게시판 관리) |
| **2** | `ROLE_STAFF_L2` | 사무직 관리자 | **GLOBAL** (운영 승인 및 통계) |
| **3** | `ROLE_ADMIN` | 최고 관리자 | **SYSTEM** (설정 및 민감 정보) |

---

## 2. 데이터 아키텍처: 테이블 분리 표준

보안과 업무 정보 관리를 위해 사용자 정보는 두 개의 테이블로 분리하여 관리합니다.

### 2.1. `users` 테이블 (기초 신상 및 인증)
법적으로 민감하거나 인증에 필수적인 정보를 담는 테이블입니다.

| 항목명 (Field)   | 데이터 타입 (Type) | 보안 등급    | 비고 (Description/Constraint)                      |
| :------------ | :------------ | :------- | :----------------------------------------------- |
| `id`          | Integer (PK)  | -        | 자동 생성 고유 번호                                      |
| `username`    | String        | PUBLIC   | 로그인 ID                                  |
| `password`    | String (Hash) | INTERNAL | 비밀번호 암호화 저장                              |
| `real_name`   | String        | PUBLIC   | 실명 (게시판 등에 표시될 공식 성함) |
| `phone`       | String        | L2 ONLY  | 개인 연락처 (휴대폰)                                     |

### 2.2. `user_profiles` 테이블 (업무 확장 및 상세 정보)
드라이버 관리, 배차, 행정 업무에 필요한 데이터를 담는 테이블입니다.

| 항목명 (Field)    | 데이터 타입 (Type) | 접근 권한   | 비고 (Description/Constraint) |
| :------------- | :------------ | :------ | :-------------------------- |
| `user_id`      | Integer (FK)  | -       | `users.id`와 1:1 관계          |
| `rank_level`   | Integer       | PUBLIC  | **권한 핵심 (0~3)**               |
| `is_active`    | Boolean       | PUBLIC  | 재직 여부 (True/False)          |
| `employee_no`  | String        | PUBLIC  | 공식 사번                          |
| `resident_no` | String (Enc)  | L3 ONLY  | 주민번호 (양방향 암호화)              |
| `joined_date`  | Date          | PUBLIC  | 입사일                         |
| `bank_name`    | String        | L2 ONLY | 급여 은행                       |
| `account_no`   | String        | L2 ONLY | 계좌 번호                       |
| `admin_memo`   | Text          | L2 ONLY | 관리자용 비공개 메모 |

## 3. 지능형 권한 제어 (Model Property)

코드의 중복을 줄이고 유지보수성을 높이기 위해 `User` 모델 내에서 실시간으로 권한을 계산합니다.

### 3.1. Role (누구인가?)
`rank_level`을 기반으로 Python `@property`를 통해 생성됩니다.
- 예: `user.role` -> `"ROLE_DRIVER"` 또는 `"ROLE_STAFF_L1"`

### 3.2. Permission & Scope (어디까지 가능한가?)
게시판 설정(`BoardConfig`)의 JSONB 필드(`perm_read`, `perm_write`)에서 나의 Role에 맞는 Scope를 찾아옵니다.
- **OWN (본인):** 본인과 관련된 데이터만 접근 가능.
- **GLOBAL (전체):** 시스템 내 해당 보드의 모든 데이터에 접근 가능.

---

## 4. 실무 적용 시나리오 예시

| 상황 | 사용자 | Rank | Role | Scope | 결과 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **게시물 읽기** | 운전원 A | 0 | `ROLE_DRIVER` | **OWN** | 본인이 쓴 글만 보임 (설정에 따라) |
| **게시물 읽기** | 팀장 B | 2 | `ROLE_STAFF_L2` | **GLOBAL** | 모든 글 목록 보임 |
| **게시물 삭제** | 운전원 A | 0 | `ROLE_DRIVER` | **OWN** | 내가 쓴 글만 삭제 가능 |
| **게시물 삭제** | 실무자 C | 1 | `ROLE_STAFF_L1` | **GLOBAL** | 부적절한 모든 글 삭제 가능 |

---

## 5. 보안 및 이행 지침

1. **최소 권한의 원칙:** 모든 사용자는 업무 수행에 필요한 최소한의 권한과 스코프만 부여받아야 합니다.
2. **지능형 모델 활용:** 서비스 로직에서 직접 `rank_level`을 체크하기보다, `user.role`이나 `user.get_scope()`와 같은 모델 메서드를 활용합니다.
3. **민감 정보 암호화:** `user_profiles` 테이블의 `resident_no`는 반드시 양방향 암호화되어야 합니다.

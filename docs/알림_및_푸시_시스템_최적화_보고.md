# 🔔 알림 및 푸시 시스템 최적화 보고서 (2026-02-27)

어제(2월 26일)까지 진행된 웹소켓 및 푸시 알림 기능을 바탕으로, 시스템의 안정성과 응답 속도를 높이기 위해 수행된 수정 사항입니다.

---

## 1. 주요 수정 요약
*   **성능 최적화**: 긴급 알림(Level 5) 생성 시 웹 푸시 발송을 **백그라운드 작업**으로 전환하여 응답 지연을 제거했습니다.
*   **구조 개선**: 푸시 발송 로직을 **`push_service.py`로 공통화**하여 코드 중복을 제거하고 관리 효율을 높였습니다.
*   **안정성 보강**: 백그라운드에서 데이터베이스 세션 충돌이 발생하지 않도록 세션 관리 로직을 보완했습니다.

---

## 2. 상세 수정 내용

### ① `domain/alert/alert_router.py` & `alert_crud.py`
*   **수정 이유**: Level 5 알림은 모든 사용자에게 푸시를 보냅니다. 구독자가 많을 경우 푸시 발송이 끝날 때까지 관리자 화면이 멈춰있는(응답 대기) 문제를 해결해야 했습니다.
*   **수정 방법**: FastAPI의 `BackgroundTasks` 기능을 도입했습니다.
*   **기존 방식 vs 수정 방식**:
    *   **기존**: 알림 저장 → **푸시 발송 완료까지 대기** → 결과 응답 (느림 🐢)
    *   **수정**: 알림 저장 → **푸시 발송을 백그라운드 예약** → 즉시 결과 응답 (빠름 🚀)

### ② `domain/push/push_service.py` (신규 및 수정)
*   **수정 이유**: 알림 도메인(`alert`)뿐만 아니라 다른 곳에서도 푸시를 보낼 수 있도록 로직을 분리하고, 백그라운드 실행 시의 세션 문제를 해결해야 했습니다.
*   **수정 방법**: `send_push_to_all` 함수가 직접 데이터베이스 세션을 열 수 있도록 보완했습니다.
*   **기존 방식 vs 수정 방식**:
    *   **기존**: API 요청의 세션을 그대로 사용. 백그라운드 실행 시 요청이 끝나 세션이 닫히면 에러 발생 위험이 있음.
    *   **수정**: 세션이 전달되지 않으면(`None`) 내부에서 새 세션을 열고 작업 완료 후 닫음. 백그라운드 작업에 안전함.

### ③ `domain/push/push_router.py`
*   **수정 이유**: 기존에 라우터에 직접 구현되어 있던 지저분한 로직과 중복된 환경 변수 설정을 정리했습니다.
*   **수정 방법**: 모든 발송 로직을 `push_service.py` 호출로 단축했습니다.
*   **차이점**: 코드가 간결해졌으며, 발송 로직 변경 시 `push_service.py` 한 곳만 수정하면 됩니다.

## 3. 결과 및 기대 효과
1.  **관리자 경험 개선**: 긴급 알림을 등록하자마자 화면이 즉시 갱신되어 다음 업무를 바로 볼 수 있습니다.
2.  **시스템 안정성**: 푸시 서버(Google/Apple 등)의 응답이 느려지더라도 전체 시스템의 웹소켓이나 API 성능에 영향을 주지 않습니다.
3.  **코드 가독성**: 기능별로 역할(Router, CRUD, Service)이 명확히 분리되었습니다.
